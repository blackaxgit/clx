## B5-4-audit — Hash-chained `validator_disabled` audit record

### Fix location

- New module: `crates/clx-hook/src/audit_chain.rs` (lines 1–298)
- Integration: `crates/clx-hook/src/hooks/pre_tool_use.rs` lines 28–76
- Module registration: `crates/clx-hook/src/lib.rs` (`pub(crate) mod audit_chain;`)
- New deps: `crates/clx-hook/Cargo.toml` — `sha2 = "0.10"`, `hex = "0.4"`

### Before behavior

`pre_tool_use.rs` loaded `Config` and immediately routed by tool name. Security-weakening environment variables (e.g. `CLX_VALIDATOR_ENABLED=false`) were silently active with no tamper-evident record. An attacker who set these variables and then unset them before forensic review left no verifiable trace.

### After behavior

Immediately after `Config::load()`, `config.security_env_overrides_active()` is called. If any of the four security-weakening CLX vars are active, a SHA-256 hash-chained `AuditChainRecord` (seq=1, genesis sentinel `GENESIS_HASH`) is built from the env-var **name(s) only** (never values, argv, or cwd). The head hash is emitted to `tracing::warn!` so it can be captured by an external append-only sink. A structured `SECURITY-ENV` row is also written to the audit log via `log_audit_entry` (which itself applies B6-3 redaction). Chain lives entirely within clx-hook; no clx-core changes required.

### Fail-before assertion

Without the hash chain, an adversary who sets `CLX_VALIDATOR_ENABLED=false`, triggers a hook, and then unsets the variable has committed a bypass with no verifiable audit trail. The only previous evidence was a plain `warn!` log line — a string that neither chains to prior events nor is detectable as tampered when replaced.

Formally: `verify_chain([tampered_record])` must return `Err`. Before this fix, no `verify_chain` existed; the bypass was undetectable.

### Adversarial re-derivation

The canonical form used for hashing is:

```
seq={seq};ts={timestamp};event={event_type};keys={trigger_keys}
```

`entry_hash = SHA-256(canonical_bytes || prev_hash_bytes)`

An adversary who wants to hide `CLX_VALIDATOR_ENABLED` from `trigger_keys` must:
1. Rewrite `trigger_keys` in the audit row (e.g. to `"TAMPERED"`).
2. Recompute `entry_hash` from the new canonical form.
3. Rewrite the audit row with the new hash.
4. Convince the external log sink (syslog / SIEM) to accept the new `head_hash` in place of the anchored warn-log line.

Step 4 is the external anchor: the `tracing::warn!(head_hash = ...)` is captured before the process exits. A local same-uid attacker can rewrite the SQLite audit row and recompute the hash, but cannot retroactively alter the append-only external sink. The chain is therefore tamper-evident (detectable with external anchor) even though it is not tamper-proof against a root/same-uid attacker who also controls the log sink.

In `crates/clx-hook/src/audit_chain.rs` `#[cfg(test)]` module:

- `single_record_chain_verifies` — happy path, one record
- `multi_record_chain_verifies` — three-record chain
- `tampered_trigger_keys_breaks_chain` — **primary evidence bundle test** (adversarial re-derivation)
- `deleted_record_breaks_chain` — gap/deletion vector
- `reordered_records_break_chain` — swap vector
- `tampered_entry_hash_breaks_chain` — direct hash flip vector
- `tampered_timestamp_breaks_chain` — timestamp rewrite vector
- `all_single_field_tamper_vectors_detected` — proptest-style table (trigger_keys, timestamp, alt-keys hash inequality, seq)
- `genesis_hash_is_all_zeros` — sentinel constant invariant
- `empty_chain_verifies` — base case

In `crates/clx-hook/src/hooks/pre_tool_use.rs` `#[cfg(test)]`:

- `b5_4_no_audit_record_without_active_overrides` — zero-overhead non-regression on the clean path

### Residual risk

- Per-invocation seq=1 (hook is a short-lived process, not a daemon). An adversary who replays the process multiple times produces independent seq=1 chains; cross-invocation ordering is not chain-verified. The external timestamp and warn-log anchor are the cross-invocation correlation mechanism.
- A same-uid attacker with SQLite write access can recompute the whole chain. The external sink anchor is the only defence beyond that.
- `security_env_overrides_active()` reads the 4 documented CLX vars. A novel security-weakening var not in that list would not trigger the chain.

## B6-3 — Redact `working_dir` and `reasoning` before audit persistence

### Fix location

`crates/clx-hook/src/audit.rs` lines 36–38

```
// Before (lines 36-38 pre-fix):
entry.working_dir = Some(working_dir.to_string());
entry.reasoning = reasoning.map(std::string::ToString::to_string);

// After:
entry.working_dir = Some(redact_secrets(working_dir));
entry.reasoning = reasoning.map(redact_secrets);
```

`command` was already passed through `redact_secrets` (line 23 — unchanged). `working_dir` and `reasoning` were stored verbatim.

### Before behavior

`working_dir` was stored raw. An agent-provided CWD could embed inline API keys (e.g. a user who accidentally used a token as a path component: `/home/user/sk-abc123LONGTOKEN456/work`). `reasoning` was the L1 model's output, which can echo command fragments including secrets. Both fields were written verbatim to the SQLite audit log.

### After behavior

Both `working_dir` and `reasoning` pass through `redact_secrets` before being stored. The scrubber is pattern-based (API key prefixes, kv-heuristic pairs, Azure tenant host suffixes) and is deliberately conservative: ordinary prose is preserved byte-for-byte so forensic value is retained for non-secret content.

### Fail-before assertion

`redact_secrets("/home/user/sk-abc123LONGTOKEN456/work")` before this fix returned the input unchanged (no call was made — the value was stored directly). After the fix, the stored value contains `***REDACTED***` and does not contain `sk-abc123LONGTOKEN456`.

### Adversarial re-derivation

An adversary who sets a CWD containing an API key (via `tool_input.cwd` in the hook envelope) could cause the audit log to persist the raw key. A downstream log export or audit log read would leak the credential. After B6-3, the scrubber runs before the `create_audit_log` call; the leaked value reaches the DB as `***REDACTED***`.

### Test names

In `crates/clx-hook/src/audit.rs` `#[cfg(test)]` module:

- `b6_3_working_dir_with_secret_is_redacted` — `sk-` token in CWD path is scrubbed
- `b6_3_reasoning_with_secret_is_redacted` — `sk-` token in reasoning string is scrubbed
- `b6_3_ordinary_prose_reasoning_is_preserved` — non-regression: prose survives unchanged
- `b6_3_benign_working_dir_is_preserved` — non-regression: real path survives unchanged
- `b6_3_azure_tenant_host_in_reasoning_is_redacted` — Azure tenant host (space-separated form) is scrubbed

### Residual risk

- The scrubber uses Pass 2 whitespace/quote tokenization. A token immediately followed by `:` (e.g. `host.openai.azure.com:`) is not a token boundary hit; the Azure host in `host.azure.com: deployment not found` would not be caught. Documented in the test comment. This is a known limitation of the B6-2 tokenizer, not introduced by B6-3.
- `redact_secrets` is pattern-based, not semantic. A novel credential format not matching any registered prefix or kv-pattern would survive unredacted. Defense-in-depth: the audit log is access-controlled at the filesystem level.

## B6-4 — Debug-log via `redact_json_value` on parsed envelope

### Fix location

`crates/clx-hook/src/router.rs` lines 232–247 (replacing old single `debug!` call)

```rust
// Before:
debug!("Hook input: {}", redact_secrets(&raw));

// After (lines 241-247):
if tracing::enabled!(tracing::Level::DEBUG) {
    let redacted_dbg = serde_json::from_str::<serde_json::Value>(&raw).map_or_else(
        |_| redact_secrets(&raw),
        |v| redact_json_value(&v).to_string(),
    );
    debug!("Hook input: {}", redacted_dbg);
}
```

Import added at line 25: `use clx_core::redaction::{redact_json_value, redact_secrets};`

### Before behavior

`redact_secrets(&raw)` was called on the raw JSON string. The free-text kv-heuristic looks for `key=value` or `key: value` patterns in the linear byte stream. JSON uses `"key":"value"` syntax — the colon is immediately followed by `"` with no space in most serializations. A secret value under a key named `password` or `token` in JSON would not be caught by the kv pattern `password: sk-...` because the actual bytes are `"password":"sk-..."`. The prefix matcher (e.g. `sk-`) would still catch values with known prefixes regardless of key, but values without known prefixes (e.g. a bcrypt-hashed password, a random UUID used as a bearer token) would be logged in clear text.

### After behavior

When DEBUG-level tracing is enabled, the raw string is first parsed as `serde_json::Value`. `redact_json_value` walks the parsed tree and applies `redact_secrets` per string leaf, plus unconditionally redacts string leaves whose parent key name matches a sensitive-key list (e.g. `password`, `token`, `secret`, `authorization`). For non-JSON input (malformed/fragment), the path falls back to `redact_secrets(&raw)` unchanged — no panic, no behavior regression. The `tracing::enabled!` gate prevents double-parse cost in non-debug builds (the macro does not evaluate its arguments when the level is filtered out; here we hoist the check explicitly since the parse is inside the arg expression).

### Fail-before assertion

Given envelope `{"session_id":"sess","cwd":"/tmp","hook_event_name":"PreToolUse","password":"hunter2-not-a-prefix"}`, `redact_secrets` on the raw string does not catch `hunter2-not-a-prefix` (no prefix match, and the kv pattern `password: hunter2-not-a-prefix` is not present — the actual bytes are `"password":"hunter2-..."`). `redact_json_value` catches it because the parent key `password` is in the sensitive-key set.

### Adversarial re-derivation

A malicious hook envelope that embeds a credential under `"token"` or `"password"` (keys that a realistic JSON protocol extension might use) would be logged in plain text with the old path. An attacker who gains read access to the debug log captures the credential. After B6-4, the structured redactor scrubs by key name regardless of value format.

### Test names

In `crates/clx-hook/src/router.rs` `#[cfg(test)]`:

- `b6_4_redact_json_value_catches_structured_secret_redact_secrets_misses` — verifies `redact_json_value` scrubs `"password"` and `"token"` keys; verifies `session_id`/`hook_event_name` preserved
- `b6_4_non_json_input_falls_back_to_redact_secrets_without_panic` — verifies the `map_or_else` fallback scrubs `sk-` prefix via `redact_secrets` and preserves non-secret text

### Residual risk

- The `tracing::enabled!(tracing::Level::DEBUG)` gate is process-wide; if a downstream log subscriber silently enables DEBUG for only certain targets, the gate passes and the parse runs. Acceptable: if DEBUG is active, the log is written; the scrubber runs either way.
- `redact_json_value`'s sensitive-key list is finite. A novel key name not in the list and whose value has no known prefix would survive. Not mitigated here; out of B6-4 scope.

## B1-10 — Remove mtime-only legacy plain-text trust-token fallback

### Fix location

`crates/clx-hook/src/hooks/pre_tool_use.rs` — the `else` branch of the trust-token JSON parse, approximately at the original lines 115–122.

```rust
// Before (pseudocode of removed branch):
} else {
    // Legacy fallback: mtime < 1h → trust_valid = true
    if let Ok(meta) = std::fs::metadata(&token_path) {
        if let Ok(modified) = meta.modified() {
            if modified.elapsed().unwrap_or(Duration::MAX) < Duration::from_secs(3600) {
                trust_valid = true;
            }
        }
    }
}

// After:
} else {
    // B1-10: mtime-only legacy plain-text trust-token fallback removed.
    // mtime is not authentication — touching a file (same-uid) granted
    // 1h global auto-allow of all commands, which is a security downgrade.
    // Only the signed JSON TrustToken (expiry + session binding) grants
    // trust. A non-JSON token file → trust_valid = false → falls through
    // to normal validation (fail-safe: more prompting, never more allowing).
    // Migration: run `clx trust` to write a proper JSON token.
    false
}
```

`CHANGELOG.md` `[Unreleased]` section updated with migration note.

### Before behavior

`~/.clx/.trust_mode_token` was first parsed as JSON (`TrustToken`). On JSON parse failure (e.g. a file containing a raw timestamp string or any non-JSON content), the code fell through to the mtime branch: if the file's modification time was within the last hour, `trust_valid = true` — granting global auto-allow for all commands. `touch ~/.clx/.trust_mode_token` by any process running as the same UID was sufficient to obtain 1 hour of unrestricted command execution.

### After behavior

On JSON parse failure, the `else` branch returns `false` unconditionally. `trust_valid` is not set. The hook falls through to normal validation (L0/L1 policy, prompting). This is fail-safe: the worst outcome of the removal is that a user who has a legacy token file gets prompted more. No valid command is silently blocked; no security constraint is weakened.

### Fail-before assertion

A file at `~/.clx/.trust_mode_token` containing the literal string `not-json` with `mtime < now - 1h` granted trust. After B1-10, the same file causes `trust_valid = false` and the hook falls through to normal validation. The integration test `test_hook_trust_mode_legacy_plain_text_token_falls_through` (in `crates/clx-hook/tests/integration.rs`) asserts `"deny"` where it previously asserted `"allow"`.

### Adversarial re-derivation

An attacker with write access to `~/.clx/` (same UID) could create any file at `~/.clx/.trust_mode_token` — even `touch` alone suffices. With the mtime fallback active, this granted 1 hour of unrestricted `auto-allow` for every Bash/MCP command, bypassing L0 rules, L1 validation, and the deny-list entirely. The attacker then unsets the file before the session ends, leaving only a brief mtime timestamp as evidence (which is also forgeable). After B1-10, creating the file with non-JSON content produces no elevated trust; only `clx trust` (which writes a properly structured `TrustToken` with cryptographic expiry binding) grants elevated trust.

### Test names

In `crates/clx-hook/src/hooks/pre_tool_use.rs` `#[cfg(test)]`:

- `b1_10_non_json_token_does_not_grant_trust` — non-JSON token file → `trust_valid = false`
- `b1_10_valid_json_token_still_grants_trust` — proper `TrustToken` JSON → `trust_valid = true` (non-regression)

In `crates/clx-hook/tests/integration.rs` (renamed):

- `test_hook_trust_mode_legacy_plain_text_token_falls_through` — end-to-end: plain-text token → decision `"deny"` (was `"allow"` before B1-10; this rename+assertion-flip is the regression guard)

In `crates/clx-hook/tests/validation_e2e.rs` (renamed):

- `trust_mode_legacy_plain_text_token_falls_through_after_b1_10` — E2E: plain-text token → `"deny"`, zero TRUST audit rows

### Residual risk

- `TrustToken` fields `enabled_at`, `expires_at`, `duration_secs`, `session_id`, `enabled_by` are validated structurally (serde deserialization) but the expiry check at runtime relies on the system clock. A system clock rollback by a same-uid attacker with `CAP_SYS_TIME` could extend an expired token. Accepted: clock manipulation is out of scope for this item.
- The migration path (`clx trust`) requires user action. Users with automated scripts that `touch ~/.clx/.trust_mode_token` will see more prompting until they migrate. The CHANGELOG migration note addresses this.
