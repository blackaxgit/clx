## Fix 1 — B5-4 + R1-NEW-1: Security-weakening env override WARN + auditable accessor

### Root cause

`crates/clx-core/src/config/mod.rs:1240–1275` (`apply_env_overrides`).

Four `CLX_VALIDATOR_*` env vars have the highest precedence and are applied unconditionally by `apply_bool_override` / `apply_enum_override`. A successful apply of a security-weakening value (validator disabled, L1 disabled, default_decision=allow, auto_allow_reads enabled) was entirely silent — no log output, no accessible signal for callers.

### Fix file:line

`crates/clx-core/src/config/mod.rs`

1. **WARN emission** — lines 1241–1275 (the four `CLX_VALIDATOR_*` apply blocks). After each successful `apply_bool_override` / `apply_enum_override` for a security-weakening value, a `tracing::warn!` is now emitted with:
   - the env var name (`env_var = "CLX_VALIDATOR_ENABLED"`)
   - the value as set
   - a human-readable description of why it weakens the validator
   - the fixed text `"Audit trail: env var weakens security posture."`

   The WARN fires only when the override actually degrades posture (i.e. `CLX_VALIDATOR_ENABLED=false` warns; `CLX_VALIDATOR_ENABLED=true` does not; `CLX_VALIDATOR_DEFAULT_DECISION=allow` warns; `=deny` or `=ask` do not; `CLX_VALIDATOR_AUTO_ALLOW_READS=true` warns; `=false` does not).

2. **`Config::security_env_overrides_active()` accessor** — new public method, lines 1211–1254 (before `expand_tilde`). Returns `Vec<(&'static str, String)>` — each entry is `(env_var_name, raw_value_string)` for every active security-weakening env var. The method is stateless (re-reads env at call time) so it is always consistent regardless of when it is called relative to `apply_env_overrides`. Callers (e.g. the hook audit log — in `clx-hook`, which is cross-ownership and not touched here) can call this to emit a structured audit row.

### WARN design rationale

- Legitimate ops use (CI, container entrypoints) will see the WARNs in their log output — acceptable, as ops staff expect WARNs when overriding security defaults.
- The WARN fires per-apply (once per process startup) — not per command — so it is not spammy.
- The message identifies the exact env var, current value, and consequence, making it grep-able from log files.

### Accessor design rationale

- Stateless re-read: the accessor does not rely on internal mutable state, so it cannot be stale or race with `apply_env_overrides`.
- Cross-ownership safety: `clx-hook` can call `config.security_env_overrides_active()` after loading config; it does not need to reach into `apply_env_overrides` internals.
- Returns an empty Vec when no weakening vars are in effect, so the normal hot path is zero overhead.
- Does not expose non-security-weakening overrides (e.g. `CLX_VALIDATOR_LAYER1_TIMEOUT_MS`) — the API surface is minimal and precise.

### Before / after

| Before | After |
|---|---|
| `CLX_VALIDATOR_ENABLED=false` applied silently by `apply_bool_override` | `tracing::warn!` emitted immediately after apply, naming the var and consequence |
| No way for callers to know which weakening vars were active | `config.security_env_overrides_active()` returns the full list |
| `CLX_VALIDATOR_DEFAULT_DECISION=allow` applied silently | `tracing::warn!` emitted identifying fail-open consequence |
| `CLX_VALIDATOR_AUTO_ALLOW_READS=true` applied silently | `tracing::warn!` emitted identifying reads bypass |

## Fix 2 — B3-1: MCP credential mask hardened (no plaintext fragment, no exact length)

### Root cause

`crates/clx-mcp/src/tools/credentials.rs:151–160` (`mask_credential_value`).

Old implementation: `first3...last3 (N chars)` — leaked 6 plaintext characters (3 head + 3 tail) plus the exact character count. For structured secrets (AWS AKIA keys, Azure keys with known prefixes/suffixes) the head+tail reduces the brute-force keyspace from >2^128 to a practical enumeration target. Exact length allows precise format fingerprinting. Reachable by an indirect prompt-injected agent with no same-uid code execution requirement.

### Fix file:line

`crates/clx-mcp/src/tools/credentials.rs:146–183`

`mask_credential_value` replaced entirely. New implementation:

```
[REDACTED:<bracket>]
```

where `<bracket>` is one of `empty`, `short` (1–15 chars), `medium` (16–63 chars), `long` (64–255 chars), `very-long` (256+).

### New mask scheme

- **No plaintext fragment**: the output string contains zero characters from the original value (no head, no tail, no interior slice).
- **No exact length**: the coarse bracket is deliberately wide (e.g. "medium" covers 16–63 chars). Two secrets of lengths 20 and 50 produce identical masked output. An adversary cannot reconstruct the exact length from the bracket.
- **Fixed form**: the output is always `[REDACTED:<bracket>]` — a machine-parseable, grepping-friendly token. No freeform suffix that could accidentally encode information.
- **Coarse bucket widths chosen to prevent enumeration**: `short` (1–15) covers PINs, short tokens; `medium` (16–63) covers API keys and passwords; `long` (64–255) covers JWTs and certificates; `very-long` (256+) covers large blobs. Width is at minimum 2× the bracket floor, preventing modular enumeration.

### Residual leak analysis

| Information class | Pre-fix | Post-fix |
|---|---|---|
| Head plaintext (first 3 chars) | Leaked | Not leaked |
| Tail plaintext (last 3 chars) | Leaked | Not leaked |
| Exact character count | Leaked | Not leaked |
| Coarse length bracket | Implicit | Explicit, coarse (16 values in range) |
| Credential existence | Yes (key name present in response) | Yes (unchanged — existence must be reported) |
| Key name | Yes (by design) | Yes (unchanged) |

Residual: a credential's existence and key name are always disclosed by a `get` call — this is necessary for the tool to be functional and is an accepted, documented behavior (B3-5 in the Track register). The coarse bracket slightly reduces brute-force keyspace compared to no length hint, but the practical impact is negligible (the bracket covers hundreds of possible lengths within each band).

### Before / after

| Value | Pre-fix output | Post-fix output |
|---|---|---|
| `AKIA-SYNTHETIC-EXAMPLE-1234-TAIL` (32 chars) | `AKI...AIL (32 chars)` | `[REDACTED:medium]` |
| `abc` (3 chars) | `**** (3 chars)` | `[REDACTED:short]` |
| `supersecret42` (13 chars) | `sup...t42 (13 chars)` | `[REDACTED:short]` |
| empty string | `**** (0 chars)` | `[REDACTED:empty]` |

## PoC disposition

### B5-4 / R1-NEW-1 (red_r1_validator_bypass.rs — `b5_4_env_overrides_win_over_project_and_global`)

The RED PoC test file exists on `chore/pre-release-rgp-hardening` (parent project), not in this worktree (`feat/0.8.0-memory-skills-coverage`). The PoC demonstrates that env vars win (this is legitimate — env precedence is preserved; the fix does not remove it). The PoC does NOT assert that the override is silent — it only asserts the config values are overridden. Since the fix does not change the override values (only adds WARNs and the accessor), the PoC test will continue to pass on the parent branch.

The GREEN fix closes the FINDING (silent application) not the MECHANISM (env precedence). PURPLE should re-run the PoC on the parent branch and confirm: (a) the env vars still win (PoC assertions hold), (b) a `tracing::warn!` subscriber would capture the WARN (testable via `tracing_test` or subscriber capture), (c) `security_env_overrides_active()` returns all four overrides.

### B3-1 (red_r2_poc.rs — `b3_1_mask_leaks_six_chars_and_exact_length`)

The RED PoC defines a local `mask_credential_value` mirror function byte-for-byte and asserts:
- `masked.starts_with("AKI")` — would now fail (output is `[REDACTED:medium]`)
- `masked.contains("ail (")` — would now fail
- `masked.contains("(32 chars)")` — would now fail
- `mask_credential_value("abc123") == "**** (6 chars)"` — would now fail (output is `[REDACTED:short]`)

All four PoC assertions are inverted by the fix. PURPLE should copy the fix into the parent branch, re-run the PoC, and confirm all four assertions fail (proving the PoC can no longer demonstrate the vulnerability). The `b3_1_poc_secret_no_longer_leaks` regression test directly inverts the first three assertions.
