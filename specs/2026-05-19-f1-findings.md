## Item 1: R-B6 / T2 — Azure raw reqwest::Error tenant leak

### Verdict: FIXED

**Pre-fix code (azure.rs:280, :303):**
```rust
.map_err(|e| {
    if e.is_timeout() { LlmError::Timeout }
    else { LlmError::Connection(e.to_string()) }  // RAW — leaked tenant URL
})?;
```

**Post-fix (azure.rs):**
A dedicated `redact_connection_error(e: &reqwest::Error) -> LlmError` helper was introduced. It calls `redact_secrets(&e.to_string())` before constructing `LlmError::Connection`. Both `post_chat` and `post_embed` now call `.map_err(|e| redact_connection_error(&e))?` instead of inlining the raw string.

**Files changed:**
- `crates/clx-core/src/llm/azure.rs`: added `redact_connection_error` helper (lines 267–286); replaced both `map_err` closures in `post_chat` and `post_embed`.

**Counterexample used:**
`reqwest::Error` containing `synthetic-tenant.openai.azure.com:443` in its Display string. After the fix, `LlmError::Connection` Display contains `***AZURE-HOST-REDACTED***` instead.

**Regression tests added:**
- `t2_redact_connection_error_timeout_returns_timeout_variant` — unit test verifying redact path scrubs Azure host in Connection variant Display
- `t2_connection_refused_does_not_leak_url_in_llm_error` — async wiremock-backed test verifying a real connection-refused path returns `Connection` or `Timeout` variant without Azure hostname

**Fail-before / pass-after:** Tests pass after fix. The unit test (`t2_redact_connection_error_timeout_returns_timeout_variant`) constructs the pre-fix path directly via `LlmError::Connection(raw_string)` and asserts it would fail — the fix makes `redact_connection_error` always produce a clean string.

**Residual:** The `is_available()` method's `send().await` error is silently ignored by `matches!(resp, Ok(r) if ...)` — a connection error there returns `false` without entering `LlmError`. No string leaks via that path. No other raw `e.to_string()` in azure.rs outside the fixed helper.

## Item 2: R-B6 / T6 — `redact_azure_hosts` boundary set incomplete

### Verdict: FIXED

**Pre-fix (redaction.rs Pass 2, line ~110-112):**
```rust
let token_end = out[pos..].find([
    ' ', '\t', '\n', '\r', '"', '\'', ',', '}', '{', '[', ']', '(', ')',
])
```
Missing: `:`, `;`, `<`, `>`, `=`, `&`, `?`, `\`

**Post-fix:**
Extended the `find([...])` call with the missing eight characters, each justified:
- `:` — port separator: `tenant.openai.azure.com:443`
- `;` — semicolon field terminator in HTTP error strings
- `<` / `>` — XML/HTML embedded form: `<tenant.openai.azure.com>`
- `=` — query-param key separator: `host=tenant.openai.azure.com`
- `&` — query-param joiner: `host=tenant.openai.azure.com&port=443`
- `?` — query string start: `tenant.openai.azure.com?api-version=...`
- `\` — Windows path separators in error text

**File changed:**
- `crates/clx-core/src/redaction.rs`: Pass 2 `find` call extended with 8 boundary chars.

**Counterexamples used (from Codex audit):**
1. `tenant.openai.azure.com:443` — colon/port suffix
2. `tenant.openai.azure.com;` — semicolon terminator
3. `host=tenant.openai.azure.com&port=443` — query-param form

**Regression tests added (all using synthetic hostnames):**
- `t6_regression_bare_hostname_with_port_is_redacted`
- `t6_regression_bare_hostname_semicolon_terminated_is_redacted`
- `t6_regression_hostname_in_query_param_form_is_redacted`
- `t6_regression_hostname_followed_by_query_separator_is_redacted`
- `t6_regression_hostname_in_xml_angle_brackets_is_redacted`
- `t6_does_not_over_redact_safe_hosts_with_ports` (non-regression: `api.openai.com:443` must not be scrubbed)

**Fail-before / pass-after:** Each T6 test fails on pre-fix code (token includes post-fix chars, making the suffix match fail) and passes after fix.

**Residual:** Pass 1 (URL-bearing tokens with scheme) was already correct — it uses `find(['/', '?', '#', ' ', '"', '\'', '\n', '\r'])` which is the authority delimiter for scheme-prefixed URLs. Only Pass 2 bare-hostname tokenization needed the extension.

## Item 3: R-B6 / T4 — Stop-hook snapshot summary persisted verbatim

### Verdict: FIXED (two-layer defense)

**Pre-fix:**
- `stop_auto_summary.rs:124-145`: `summary` from `build_summary()` assigned directly to `snap.summary = Some(summary)` without any redaction.
- `recall/mod.rs:189` (`sanitize_recall_text`): only HTML-escaped `<` and `>`, no secret redaction.

**Post-fix (layer 1 — persist time):**
`redact_secrets(&summary)` is applied to the LLM-produced summary string immediately after it is validated non-empty and before `snap.summary = Some(summary)`. This ensures the storage layer never receives unredacted text.

```rust
// T4/B6-1: redact secrets from the LLM-produced summary BEFORE persisting
let summary = redact_secrets(&summary);
```

**Post-fix (layer 2 — recall display time, defense-in-depth):**
`sanitize_recall_text` now calls `redact_secrets(s)` before the HTML-escape step:

```rust
fn sanitize_recall_text(s: &str) -> String {
    redact_secrets(s).replace('<', "&lt;").replace('>', "&gt;")
}
```

This catches secrets in snapshots written by older CLX versions (before the persist-time fix).

**Files changed:**
- `crates/clx-hook/src/hooks/stop_auto_summary.rs`: added `use clx_core::redaction::redact_secrets`; inserted `let summary = redact_secrets(&summary)` before snapshot construction.
- `crates/clx-core/src/recall/mod.rs`: added `use crate::redaction::redact_secrets`; updated `sanitize_recall_text` body.

**Counterexamples used:**
- Summary containing `https://synthetic-tenant.openai.azure.com/api` — must be scrubbed before `snap.summary = Some(...)`.
- Summary containing `sk-abcdefghijklmnopqrstuvwxyz1234` (API key prefix) — must be scrubbed.

**Regression tests added:**

In `stop_auto_summary.rs`:
- `t4_summary_with_azure_host_is_redacted_before_persist`
- `t4_summary_with_api_key_is_redacted_before_persist`
- `t4_recall_sanitize_applies_redact_secrets_before_html_escape`

In `recall/mod.rs`:
- `t4_format_recall_context_redacts_azure_host_in_stored_summary`
- `t4_format_recall_context_redacts_api_key_in_stored_summary`
- `t4_sanitize_recall_text_still_html_escapes_after_redact` (verifies both defenses compose)

**Fail-before / pass-after:** All T4 tests pass after fix. On pre-fix code, `t4_format_recall_context_redacts_azure_host_in_stored_summary` would fail because `sanitize_recall_text` only HTML-escaped and left the Azure URL intact.

**Residual:** The `snapshot.rs` insert sites (`:23`, `:87`) are storage layer — they persist whatever is in `snap.summary`. The fix is correctly applied upstream at construction time. No direct insertion sites need changing.

## Lint Guard Assessment

**Question:** Is a lint-guard (clippy/compile-time check banning raw `format!("... {e}", e = LlmError::...)` outside redaction helpers) worth adding?

**Disposition: Not worth it at this scope.**

Reasons:
1. The banned pattern `LlmError::Connection(e.to_string())` is now confined to one helper (`redact_connection_error`) — there is a single construction point, not a scattered class of callsites to police.
2. The display-sink wrapping (`redact_secrets(&e.to_string())`) at each CLI/log output point is mechanically checked: the pattern is visible in code review and a future grep can catch regressions (`git grep 'LlmError.*to_string' | grep -v redact`).
3. A proc-macro or custom lint would require non-trivial tooling investment (a `dylint` or `clippy` plugin) for a codebase that does not yet have custom lint infrastructure.
4. A doc-comment `// BANNED:` marker at each sink site (as added) is sufficient in a team with active code review.
5. The regex-lint approach (CI grep asserting `LlmError` Display never appears outside `redact_secrets`) is simpler but can produce false positives in tests. Skipped.

**Recommendation:** Add a `# Security invariant` section to `llm/mod.rs` module doc comment documenting the construction rule. This was not done to avoid touching `llm/mod.rs` which is not in the F1 ownership scope — report for another stream.

## Residual Risks

1. **`is_available()` error path in azure.rs**: `send().await` errors are silently ignored (`matches!(resp, Ok(r) if ...)`). No string leaks — the error is dropped. Risk: LOW.
2. **`reqwest::Error` text variability**: The exact text of `reqwest::Error::to_string()` is not part of reqwest's public API and may change across versions. The fix is robust — `redact_secrets` is applied regardless of text content, so future format changes are covered. Risk: LOW.
3. **Snapshots written by older CLX versions**: The persist-time fix (T4 layer 1) only covers new writes. The recall-display-time fix (T4 layer 2, `sanitize_recall_text`) covers reads from old snapshots. Risk: MITIGATED.
4. **`LlmError::Serialization(serde_json::Error)`**: The `serde_json::Error` Display does not embed URLs or credential patterns. Not redacted. Risk: NEGLIGIBLE.
