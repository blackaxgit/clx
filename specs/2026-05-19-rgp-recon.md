# CLX Pre-Release Threat Model & Attack-Surface Map

Date: 2026-05-19
Scope: read-only static recon of `/Users/blackax/Projects/clx` (branch `main`, HEAD `11da64b`)
Purpose: ground a red / green / purple assessment. No source modified, no PoC executed, no secret values reproduced.

Method: STRIDE per trust boundary. Each finding ranked `exploitability x impact` → severity (CRIT / HIGH / MED / LOW). File:line refs are exact. "Existing mitigations" sections tell green/purple what is already covered so they do not re-prove it.

---

## 0. Trust Model Summary

| Boundary | Untrusted input | Trusted assumption | Authentication? |
|---|---|---|---|
| B1 Command hook | stdin JSON envelope, command string, transcript path | "Claude Code spawned us" | NONE (forgeable — see F7 comment, `router.rs:73-99`) |
| B2 Credential store | local FS (`~/.clx`), env, project `.clx` | same-uid is trusted | OS file perms only |
| B3 MCP server | JSON-RPC tool args over stdio | Claude Code is the only client | NONE |
| B4 Config trust | project `.clx/config.yaml` | hostile repo is the threat | content-hash trustlist |
| B5 Supply chain / build | Cargo deps, GH Actions, env vars | maintainer + GitHub | repo/secret-based |

Core insight: CLX is a *same-uid local* security tool. Its threat model explicitly accepts that a same-uid attacker can forge hook stdin (`router.rs:76-99`). The real attack surface is therefore (a) a **malicious project repo** acting through config / prompt / transcript paths, and (b) **logic bypasses** that let a dangerous command slip the validator, and (c) **secret hygiene** because a real Azure key+tenant leaked previously.

- Cache key uses full string concat, not a hashed key — no hash-collision pre-image (`cache.rs:151-158`).

- Trust JSON token: expiry + session-id binding (`pre_tool_use.rs:78-112`).

### B2 existing mitigations
- `validate_key` allowlist + `..` block + null/length bounds (`credentials.rs:580-618`).
- Atomic write (temp + `create_new`/rename), 0600 on temp from open, dir 0700 (`backend.rs:199-326`).
- Zero-byte = corruption, never auto-destroy (`backend.rs:245-290`).
- Inter-process advisory lock w/ bounded timeout + RAII unlock on panic/exit (`backend.rs:103-117, 328-396`).
- Secrets are `SecretString` (zeroize on drop); `Debug` impls redact; session cache never serialized/logged (`credentials.rs:89-134, 430-437`).
- Keychain repair: single `/usr/bin/security` call, no password on argv, `unsigned:` partition (`keychain_acl.rs:158-184`; tests `594-639`).

### B2 — Red team probe first
1. **B2-4**: project-path scope confusion via `:` in cwd → cred-confusion / global override.
2. **B2-1**: confirm (do not exfiltrate) that `cred.key`+`credentials.age` are both same-uid readable and document the blast radius for the report (it is the leaked-Azure-key root cause class).

## B6 — Secret-Handling Hygiene (cross-cutting)

Context: a prior incident leaked an Azure API key plus its tenant URL. The real value is NOT reproduced here; all examples use the synthetic placeholder `synthetic-tenant.openai.azure.com`. Goal: enumerate every path that could log/persist a credential or tenant URL.

| ID | Finding | file:line | Risk | Sev |
|---|---|---|---|---|
| B6-1 | **Azure error bodies are surfaced verbatim into `LlmError`.** `post_chat`/`post_embed` build `format!("{body} (x-request-id: {request_id})")` from the raw HTTP response body and return it in `LlmError::Auth/Server/DeploymentNotFound/ContentFilter`. Azure 401/403/404 bodies frequently echo the deployment/endpoint and sometimes the submitted `api-key` header context or request URL. These `LlmError`s propagate up and are logged via `tracing` and rendered in hook/CLI output. **This is the most likely re-leak path for the tenant URL class.** | `llm/azure.rs:~200-260` (error mapping `format!("{body} (x-request-id...)")`, returned as `LlmError::Auth/Server/...`) | Tenant/endpoint and provider error context written to logs / Claude Code output without passing `redact_secrets`. | HIGH |
| B6-2 | **`redact_secrets` does NOT redact bare hostnames/tenant URLs.** It targets `sk-/ghp_/...` prefixes, `key=/token=/...` kv, Bearer/Basic, and `export SENSITIVE=`. A tenant URL like `https://<tenant>.openai.azure.com` matches no pattern → passes through any sink that uses it. The Azure `endpoint` is plain config, not a `SecretString`. | `redaction.rs:20-272`; `azure.rs:19` (`endpoint: Url`, not secret) | Tenant URL is not a "secret" to the redactor; B6-1 output is not scrubbed. | HIGH |
| B6-3 | **Audit log redacts command but not `reason`.** `log_audit_entry` runs `redact_secrets(command)` (good) but `entry.reasoning` is stored unredacted (`audit.rs:33`). L1 `reason` is `[{category}] {llm_reasoning}` — if the model echoes part of the command (incl. an inline secret) into its reasoning, it persists unredacted to the audit DB. | `clx-hook/src/audit.rs:22-35`; reason origin `llm.rs:551-571`, `pre_tool_use.rs:461-519` | Inline secret in a command echoed by the LLM into `reasoning` → persisted plaintext in audit log. | MED |
| B6-4 | **Hook stdin debug-logs the raw envelope (redacted).** `router.rs:232` logs `redact_secrets(&raw)` before parse — good, but only the free-text redactor (B6-2 limitations apply: a JSON-structured secret under a non-keyword field, or a tenant URL, survives). `redact_json_value` exists and is stronger but is NOT used on the raw stdin debug line (it parses later). | `router.rs:232`, `redaction.rs:329-346` | Structured/tenant data in the envelope reaches debug logs unredacted when log level is debug. | MED |
| B6-5 | Provider/credential values are `SecretString` end-to-end (zeroize, redacted Debug); MCP `get` masks; `set` never logs value; cred index/cache never serialized to logs. Good. | `credentials.rs:89-134`; `azure.rs:12,20` (`api_key: SecretString`, `expose_secret()` only at the `api-key` header) | n/a | — (mitigated) |

### B6 — Red team / purple probe first
1. **B6-1 + B6-2 together** are the concrete re-leak path for the previously-leaked tenant URL class: trigger an Azure auth/deployment error and confirm whether the endpoint/tenant appears in `tracing` output or Claude Code surface unredacted. **This is the top secret-hygiene priority.**
2. **B6-3**: command with an inline secret that the L1 model echoes into `reasoning` → check the audit DB.

### Notes & Caveats
- Severities are static-analysis estimates; dynamic confirmation is the red team's job. Several "HIGH"s (B1-7, B2-1, B2-5) are *in-model accepted risks* — flagged for completeness, not as defects.
- The dominant theme: CLX's same-uid local-trust model is internally consistent, but **B4-1 breaks it** — a hostile *repo* (not a same-uid process) neuters the validator with no user action, which is outside the accepted model.
- No secret values, tenant URLs, or PoC payloads are reproduced in this document by design.
