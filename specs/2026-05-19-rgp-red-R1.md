## 2. Confirmed-but-in-model (NOT release-blocking on their own)

### R1-06 / B4-2 — HIGH (subsumed by R1-01)

`validator.trust_mode` is **not** in the denylist — an untrusted repo sets
`trust_mode:true` (confirmed: `b4_1_untrusted_project_config_guts_validator_via_public_load`
asserts `cfg.validator.trust_mode == true` after `Config::load()` of a
hostile repo). Standalone it is **not** a full bypass: `pre_tool_use.rs:73-148`
still requires a valid `.trust_mode_token` (JSON: expiry+session bound;
legacy: mtime<3600s — R1-07). But it *widens* B1-10: a hostile repo
pre-arms trust_mode so any same-uid `touch ~/.clx/.trust_mode_token`
(R1-07) yields 1h global auto-allow. **Fix = same as R1-01** (strip
`validator.*` from untrusted configs ⇒ closes B4-2 for free).
Root cause: `config/project.rs:75-79`. CVSS as a *widener*:
`CVSS:4.0/AV:L/AC:L/AT:P/PR:L/UI:P/VC:N/VI:H/VA:N/SC:H/SI:H/SA:N` ≈ 7.1.
SSVC **Track** standalone, **Act** as part of R1-01.

### R1-07 / B1-10 — MED (in-model accepted, but cheap to harden)

Legacy plain-text trust token = **mtime-only** ≤3600s
(`crates/clx-hook/src/hooks/pre_tool_use.rs:116-122`). A same-uid
`touch ~/.clx/.trust_mode_token` (with non-JSON content, and
`validator.trust_mode` true — default false, but R1-06 supplies it from a
hostile repo) grants 1h global auto-allow, audited only as `TRUST`. This is
**in the same-uid model** for the touch itself, but the *carrier*
(R1-06 hostile-repo-set trust_mode) is out-of-model. Confirmed by code path
(no separate PoC: the mtime branch is unconditional fallback when JSON
parse fails; trivially true). CVSS
`CVSS:4.0/AV:L/AC:L/AT:N/PR:L/UI:N/VC:N/VI:H/VA:H/SC:H/SI:H/SA:N` ≈ 7.0;
SSVC **Track** (same-uid) → **Attend** when chained with R1-06.
Fix direction: remove the legacy mtime fallback entirely (require the
signed JSON token; expiry+session already implemented and tested).

### R1-08 / B1-9 — MED (in-model, documented best-effort)

`resolve_command_paths` (`crates/clx-hook/src/embedding.rs:85-112`)
`split_whitespace()` → canonicalize `/`-prefixed tokens → **rejoin with
single space**. The validator-visible string diverges from the
shell-executed string two ways, both confirmed by PoC: (1) whitespace
collapse (`echo  a   b` → `echo a b`); (2) a symlinked absolute path is
canonicalized to its target pre-L0 so L0/audit judge the *target* while the
command *names* the link. Doc explicitly admits "full TOCTOU prevention
requires Claude Code changes" → **in-model accepted**, flagged for
completeness. CVSS
`CVSS:4.0/AV:L/AC:H/AT:P/PR:L/UI:N/VC:N/VI:L/VA:L/SC:N/SI:L/SA:N` ≈ 4.8;
SSVC **Track**. PoC: `crates/clx-hook/src/embedding.rs` ::
`embedding::tests::red_r1_b1_9_resolve_paths_view_divergence`
(`cargo test -p clx-hook --lib red_r1_b1_9 -- --ignored`).
Fix direction (low priority / accepted): if hardening, match L0 on a
shell-tokenized (`shlex`) argv rather than a whitespace re-join, and never
let canonicalization shrink the audited string (audit the original AND the
resolved).

### R1-09 / B5-3 — MED (in-model env-injection, asymmetric with Ollama)

`CLX_ALLOW_AZURE_HOSTS` (`crates/clx-core/src/llm/azure.rs:54-64`) does an
exact `host == h` allow with **no `is_private_or_internal` recheck** —
unlike the Ollama remote path which re-blocks RFC1918/link-local/metadata
even under its override. Setting `CLX_ALLOW_AZURE_HOSTS=169.254.169.254`
(cloud metadata) or an internal IP fully bypasses the SSRF suffix gate.
Confirmed by reading `validate_host` (the env branch returns `Ok(())` on
exact match before any internal-IP check). **In-model**: env is the
documented injection surface (same class as R1-02); the Azure client only
issues the configured chat/embed calls (no attacker-chosen path), so
impact is "outbound request to an internal/metadata host with the Azure
api-key header" — meaningful but bounded; no runtime PoC (would require
network / a live endpoint, out of scope). CVSS
`CVSS:4.0/AV:L/AC:L/AT:P/PR:L/UI:N/VC:L/VI:N/VA:N/SC:L/SI:N/SA:N` ≈ 5.1;
SSVC **Track**. Fix direction: mirror the Ollama path — after an allowlist
hostname match, still run `is_private_or_internal` on the resolved host
unless a separate explicit "I really mean an internal Azure host" opt-in
is set; never blanket-trust an allowlisted host's IP class.

## 4. New R1-cluster findings beyond recon

- **R1-NEW-1 (folds into R1-02): `apply_bool_override` is silent on a valid
  security-weakening value.** Recon B5-4 noted the precedence; it did **not**
  note that `config/mod.rs:1878-1890` emits a WARN *only on a parse error* —
  a valid `CLX_VALIDATOR_ENABLED=false` produces **no log and no audit
  event**. This removes the only forensic trail for an env-driven disable
  and is the reason R1-02 is scored Act rather than Track. Captured in the
  R1-02 fix direction (loud warn + dedicated audit event on any
  posture-weakening env).
- **R1-NEW-2 (folds into R1-01): `validator.layer1_timeout_ms` is an
  untrusted-config L1-kill primitive.** Recon B4-1 enumerated
  `layer1_enabled`/`default_decision`/etc.; it did not call out that an
  untrusted repo setting `layer1_timeout_ms: 1` makes the L1 future at
  `pre_tool_use.rs:364-400` time out on essentially every call → the
  timeout branch applies `default_decision`. Combined with
  `default_decision: allow` this is an *alternative* full L1 bypass that
  does not even need `layer1_enabled:false` (harder to spot in a config
  review). Confirmed by `b4_1_full_settable_key_matrix`
  (`cfg.validator.layer1_timeout_ms == 1`). Closed by the same R1-01 fix
  (strip the whole `validator.*` subtree).
