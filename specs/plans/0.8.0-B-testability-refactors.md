# CLX 0.8.0 Group B: Testability Refactors

**Date:** 2026-05-16
**Branch:** `feat/0.8.0-memory-skills-coverage`
**Design doc:** `specs/2026-05-05-0.8.0-memory-skills-coverage-design.md` (sections 3.7, 3.8, 3.10, 4)
**Status:** Approved, ready for implementation
**Scope:** Three sub-phases (B1, B2, B3) that make CLX's TUI and hook binary testable without a terminal or a real Claude Code session. Net effect: lift two near-uncovered files (`event.rs` 0%, `main.rs` 67%) into the 85%+ band and lock the hook JSON envelope contract.

---

## 1. Goals and Non-Goals

### Goals

1. Make the hook event router (`clx-hook::handle_event`) testable end-to-end with in-memory readers and writers and real fixture envelopes.
2. Make the dashboard event loop testable as a pure reducer: `(state, event) -> (state', cmds)`. Tests must not require a TTY.
3. Lock the Claude Code hook stdin contract with snapshot tests. Any schema drift (theirs or ours) must fail loudly.
4. Preserve the four-layer invariant from `CLAUDE.md`: Orchestration -> Domain -> Infrastructure -> Mapping. No new cross-layer leaks.

### Non-goals (deferred)

- Refactoring downstream hook handlers (`handle_pre_tool_use`, `handle_post_tool_use`, etc.). They are already covered by unit tests at the handler level; only the router changes.
- Replacing `crossterm` or `ratatui`. Pure rendering test coverage uses `ratatui::TestBackend`.
- Mocking `tokio::runtime` itself. The runtime stays in `main()`, outside the reducer.
- Eliminating `Arc<Mutex<...>>` in `AppRuntime`. Concurrency primitives remain in the runtime half of the split.

### Layering invariants

| Layer | Belongs in |
|-------|------------|
| Orchestration | `clx-hook/src/main.rs::handle_event`, `clx/src/dashboard/event.rs::update` |
| Domain | New pure types: `HookDeps`, `AppState`, `DashboardEvent`, `DashboardCmd` |
| Infrastructure | `tokio` runtime, `crossterm::event::poll`, `ratatui::DefaultTerminal`, `Storage`, `Config` loaders |
| Mapping | `clx-hook/src/output.rs` (already isolated), JSON parse via `serde_json` |

No new dependency arrow may cross from Orchestration directly to Infrastructure without going through Domain or Mapping. All three sub-phases respect this.

## 3. Sub-phase B1: `handle_event` Refactor

**Target file:** `crates/clx-hook/src/main.rs` (190 LoC, 67% line coverage).
**Goal:** Extract a pure async router `handle_event<R: Read, W: Write>(reader, writer, deps) -> ExitCode` and shrink `main()` to ~15 LoC of wiring. Add `assert_cmd` integration tests that exercise the binary end-to-end with realistic JSON envelopes.

### 3.1 File changes

**New files:**

- `crates/clx-hook/src/router.rs` (~120 LoC) - hosts `handle_event`, `HookDeps`, `read_input`, `write_output`. New top-level module declared in `main.rs`.
- `crates/clx-hook/tests/integration/router_smoke.rs` (~180 LoC) - `assert_cmd` tests, one per hook event.
- `crates/clx-hook/tests/integration/mod.rs` (~10 LoC) - module declaration glue.

**Modified files:**

- `crates/clx-hook/src/main.rs` (190 -> ~25 LoC) - keep `print_usage`, `MutexFile`, and a thin `main()` that builds deps and calls `router::handle_event`.
- `crates/clx-hook/Cargo.toml` - add `assert_cmd = "2"` and `predicates = "3"` under `[dev-dependencies]` if not present.

### 3.2 Public API of the new module

```rust
// crates/clx-hook/src/router.rs

pub struct HookDeps {
    pub config: clx_core::config::Config,
    pub storage: clx_core::storage::Storage,
    // Logging is initialized once in main() and is process-global, so it is
    // NOT a member of HookDeps. Tests rely on tracing's default no-op subscriber.
}

pub enum HookExit {
    Ok,
    BlockedAtLimit,
    ParseError,
}

pub async fn handle_event<R, W>(reader: R, writer: W, deps: HookDeps) -> HookExit
where
    R: std::io::Read,
    W: std::io::Write;
```

`handle_event` orchestrates exactly four steps:

1. **Read** stdin into a bounded `String` (cap at `MAX_INPUT_SIZE`).
2. **Parse** to `HookInput` via `serde_json::from_str`. On error, write the existing `output_decision("ask", ...)` fallback to `writer` and return `HookExit::ParseError`.
3. **Route** to the matching handler in `hooks::*`. Each handler keeps its current signature.
4. **Write** results. Handlers already call `output::*` which uses `println!`; in B1 we leave that as-is. The `writer` parameter is reserved for the parse-error fallback path (which currently uses `println!` too). A follow-up (B1.x, not in scope here) can plumb the writer through `output::*` to make stdout itself injectable. Documented as a known limitation in the module docstring.

### 3.3 `main()` after the refactor

```rust
fn main() -> std::process::ExitCode {
    // 1. Handle --help and TTY-detection short-circuits (~5 LoC)
    // 2. Build tracing layers and init (existing block, extracted to fn init_tracing())
    // 3. Build deps and call handle_event
    let rt = tokio::runtime::Runtime::new().expect("tokio runtime");
    let deps = match build_deps() {
        Ok(d) => d,
        Err(e) => { eprintln!("CLX: init error: {e}"); return std::process::ExitCode::SUCCESS; }
    };
    let _ = rt.block_on(router::handle_event(std::io::stdin(), std::io::stdout(), deps));
    std::process::ExitCode::SUCCESS
}
```

Three new helpers in `main.rs`: `init_tracing()`, `build_deps()`, `is_help_invocation(args)`. Each under 25 LoC. The body of `main()` ends up around 15 LoC of pure wiring.

### 3.6 Risks and rollback

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Behavior drift on the parse-error fallback path (currently `output_decision("ask", ...)`) | Medium | Hooks falsely block tool use | Snapshot the existing stdout for a malformed-input run BEFORE refactor (record-only test). Refactor must reproduce identical stdout |
| `tokio::runtime` startup is moved around, breaks somewhere | Low | Hook hangs at process start | Keep the runtime constructed in `main()` only; `handle_event` is async but does not own the runtime |
| Removing the `#[tokio::main]` macro changes panic behavior in async paths | Low | Different stack trace on panic | Manual runtime + `block_on` is the documented escape hatch; existing `error!` calls still fire |
| `assert_cmd` flakes on macOS CI due to slow temp-dir cleanup | Low | Spurious test failures | Use `tempfile::TempDir` with explicit `.close()` and a 5s timeout on each subprocess; mark tests `#[ignore]`-able via `CLX_SKIP_HOOK_INTEGRATION` env var |

**Rollback strategy:** every step is committed atomically per `CLAUDE.md`. Reverting B1 is `git revert <range>`. The refactor introduces no schema, no migration, no config key, and no public crate API change. Worst case: revert and the binary is byte-identical to its pre-B1 state.

## 4. Sub-phase B2: Dashboard Reducer

**Target file:** `crates/clx/src/dashboard/event.rs` (411 LoC, 0% line coverage).
**Goal:** Convert the imperative `run_event_loop` + `handle_key_event` mutation cluster into an Elm-style reducer `update(state, event) -> (state', cmds)`. Split `App` into `AppState` (pure data, `Clone`) and `AppRuntime` (terminal handle, mutexes, timers).

### 4.1 File changes

**New files:**

- `crates/clx/src/dashboard/state.rs` (~250 LoC) - `AppState`, `DashboardEvent`, `DashboardCmd`, `update()`, `handle_*` reducer helpers. All pure.
- `crates/clx/src/dashboard/runtime.rs` (~150 LoC) - `AppRuntime` wrapping `DefaultTerminal`, `last_refresh: Instant`, `refresh_interval`. Hosts the new I/O-bearing `run_event_loop(runtime, state) -> io::Result<()>`.
- `crates/clx/src/dashboard/state_tests.rs` (~400 LoC, behind `#[cfg(test)]`) - 35+ reducer unit tests.

**Modified files:**

- `crates/clx/src/dashboard/event.rs` (411 -> ~80 LoC) - keep `handle_key_event` as a thin shim that converts `KeyEvent` to `DashboardEvent` and delegates to `state::update`. Remove direct `app.foo = bar` mutations.
- `crates/clx/src/dashboard/app.rs` (~720 LoC, mixed mutable methods on `App`) - keep `App` as the runtime-bound container. Mark all "pure" methods (`next_tab`, `prev_tab`, `cycle_sort_column`, `scroll_*`, etc.) as forwarding to `AppState` equivalents. Existing tests stay green.
- `crates/clx/src/dashboard/mod.rs` - re-export `state::AppState`, `state::DashboardEvent`, `state::DashboardCmd`, `state::update`.

### 4.2 Type design

```rust
// crates/clx/src/dashboard/state.rs

#[derive(Debug, Clone)]
pub struct AppState {
    pub current_tab: DashboardTab,
    pub should_quit: bool,
    pub input_mode: InputMode,
    pub screen_state: ScreenState,
    pub filter_text: String,
    pub sessions_sort_column: usize,
    pub sessions_sort_ascending: bool,
    pub audit_sort_column: usize,
    pub audit_sort_ascending: bool,
    pub rules_scroll_offset: u16,
    // Selection state lives in plain usize, not TableState (which is non-Clone-friendly).
    pub sessions_selected: Option<usize>,
    pub audit_selected: Option<usize>,
    pub detail_tab: DetailTab,
    pub detail_commands_selected: Option<usize>,
    pub detail_events_selected: Option<usize>,
    pub detail_snapshots_selected: Option<usize>,
    pub detail_scroll_offset: u16,
    // Settings tab data only; no Instant fields (those live in AppRuntime).
    pub settings_section_idx: usize,
    pub settings_field_idx: usize,
    pub settings_is_dirty: bool,
    pub settings_edit_buffer: String,
    pub settings_edit_error: Option<String>,
    pub settings_save_result: Option<String>,
    pub settings_confirm_reset: bool,
    pub settings_exit_pending: Option<ExitTarget>,
    pub settings_reload_confirm: bool,
    pub settings_load_error: Option<String>,
    // Snapshot of last fetched data (immutable for the lifetime of one frame).
    pub data: super::data::DashboardData,
    pub detail_data: Option<super::data::SessionDetailData>,
    pub days_filter: u32,
    pub settings_original_config: Option<clx_core::config::Config>,
    pub settings_editing_config: Option<clx_core::config::Config>,
}

#[derive(Debug, Clone)]
pub enum DashboardEvent {
    Key(crossterm::event::KeyEvent),
    Resize(u16, u16),
    Tick, // wall-clock refresh tick
}

#[derive(Debug, Clone)]
pub enum DashboardCmd {
    FetchDashboardData,
    FetchSessionDetail(String),
    SaveSettings,
    ReloadSettings,
    Quit,
}

pub fn update(state: AppState, event: DashboardEvent) -> (AppState, Vec<DashboardCmd>) { ... }
```

Key invariants:

- `AppState` is `Clone` and free of `Instant`, `Arc<Mutex<...>>`, and `TableState`.
- `TableState` is rebuilt from `*_selected: Option<usize>` inside `ui::render` each frame (existing render code already constructs frame-local widgets).
- `AppRuntime` owns the wall-clock state (`last_refresh: Instant`, `refresh_interval: Duration`) and the `DefaultTerminal`. The reducer never reads system time.
- `Tick` events are emitted by `AppRuntime` based on its own clock; the reducer treats them as opaque "data may be stale" signals.

| Claude Code schema changes mid-release | Low | All 7 parse tests fail simultaneously | This is the feature, not a bug. The CI failure announces the upstream change; we re-capture fixtures and update snapshots in a follow-up PR. Document the playbook in `CONTRIBUTING.md` |
| `insta::assert_json_snapshot!` requires JSON-serializable output but `HookInput` is `Deserialize`-only | Low | `Debug` snapshot only for parse side | We accept this asymmetry. Parse-side tests use `assert_debug_snapshot!`; emit-side tests use `assert_json_snapshot!`. Both catch the relevant class of drift |
| The `CLX_HOOK_LOG_RAW` env var leaks user data to disk | Medium | Same as fixture sanitization risk | Default off. Documented as a dev tool. File path scoped to `~/.clx/logs/`. Auto-rotate at 10 MB |

**Rollback strategy:** B3 is purely additive (fixtures + tests + one env var). Reverting deletes test files and the conditional logging block. Zero impact on production behavior.

## 6. Sequencing and Parallelization

B1, B2, B3 can run **in parallel** because their file scopes are disjoint:

| Sub-phase | Files touched |
|-----------|---------------|
| B1 | `clx-hook/src/main.rs`, `clx-hook/src/router.rs`, `clx-hook/tests/integration/` |
| B2 | `clx/src/dashboard/{event,app,state,runtime}.rs`, `clx/src/dashboard/state_tests.rs` |
| B3 | `clx-hook/tests/fixtures/`, `clx-hook/tests/contract/` |

B1 and B3 both touch `clx-hook/Cargo.toml` (dev-deps). One commit adds both `assert_cmd`, `predicates`, and `insta` simultaneously at the start of B1, avoiding a conflict.

**Recommended sequencing per CLAUDE.md "max parallelization" rule:**

```
T+0d   Pre-flight + Cargo.toml dev-dep update     (main session)
T+0d   B1 starts                                  (rust-subagent-1)
T+0d   B2a (AppState extraction, no semantic)     (rust-subagent-2)
T+0d   B3 fixture capture + parse tests           (test-automator-subagent)

T+1d   B1 merges
T+1d   B3 emit tests + CI smoke command           (test-automator-subagent)

T+2d   B2a merges; B2b (reducer + runtime) starts (rust-subagent-2)
T+2d   B3 merges

T+3d   B2b in review

T+4d   B2b merges
```

Total elapsed: 4 days with three parallel subagents.
