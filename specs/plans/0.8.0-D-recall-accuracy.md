## 0. Goals and Non-Goals

### Goals

1. Replace the linear-weighted `hybrid_merge` in `crates/clx-core/src/recall.rs:346` with **Reciprocal Rank Fusion** (`k = 60`, Cormack et al. 2009).
2. Add an **optional cross-encoder rerank stage** using `bge-reranker-v2-m3` via `fastembed-rs 4.x`, with a 250 ms per-query budget and graceful degradation to RRF-only on timeout, model-missing, or error.
3. Apply **multiplicative time-decay** (`exp(-ln(2) * age_days / 30)`) followed by a **p70 percentile gate** to keep only high-confidence hits.
4. Ship a **RAGAS-style synthetic golden set** (50 prompt/expected-snapshot pairs sourced from public CLX specs and GitHub issues) plus a `cargo bench` that computes context-precision / context-recall and emits a warn-only CI signal.

### Non-Goals (deferred)

- Reranker fine-tuning (use stock weights).
- HyDE / query rewriting (rejected for current corpus size; revisit in 0.9.0).
- Real-session golden set (synthetic only in 0.8.0 per user decision).
- Hard-fail CI gate for accuracy regressions (warn-only in 0.8.0; flip in 0.8.1).
- Daemon-mode warm cache for the reranker model (revisit when hook daemon ships).

### Layering Invariants

- `crates/clx-core/src/recall/rerank.rs` is **domain layer**. The only IO it performs is through the `RerankBackend` trait. The `fastembed-rs` adapter that loads the actual model lives behind that trait so the domain stays pure.
- `clx model fetch` lives in the **orchestration layer** (`crates/clx/src/commands/model.rs`) and is the only place that knows about HuggingFace URLs, disk paths, and progress bars.
- Recall pipeline composition (`semantic -> RRF -> rerank -> decay -> gate`) happens inside `RecallEngine::query` in `recall.rs`; the engine remains the only orchestrator of stages.

### Total Latency Budget (p95)

| Stage                                    | Budget   | Source                          |
| ---------------------------------------- | -------- | ------------------------------- |
| Stage 1: parallel embed + FTS5           | 200 ms   | existing 0.7.x behaviour        |
| Stage 2: RRF merge                       | 5 ms     | pure CPU, list-of-50 zip        |

## D1. Reciprocal Rank Fusion Replaces Linear Merge

### Files to modify

- `crates/clx-core/src/config/defaults.rs` (or equivalent) — wire defaults.
- `docs/CONFIGURATION.md` — document new keys.

### Algorithm

For each ranker `r` in `{semantic, fts5}` and each candidate document `d`:

```text
rrf_score(d) = sum_r [ 1.0 / (k + rank_r(d)) ]
```

where `rank_r(d)` is the 1-indexed rank of `d` in ranker `r`'s output. Docs missing from a ranker contribute 0 to the sum. RRF is parameter-free relative to score scales: only the rank matters, so the existing 0.6 / 0.4 weighting becomes obsolete.

### New `hybrid_merge` signature

```rust
fn hybrid_merge(
    semantic_hits: Vec<RecallHit>,
    fts_hits: Vec<RecallHit>,
    max_results: usize,
    rrf_k: u32,
    rrf_enabled: bool,
) -> Vec<RecallHit>
```

When `rrf_enabled == false`, the function delegates to a new `legacy_linear_merge` helper (kept for one release as a rollback path). When `true`, it computes RRF scores, sorts descending, and truncates to `max_results`. The `search_type` field for merged hits is set to `RecallSearchType::Hybrid` whenever both rankers contributed; otherwise the original ranker's variant is preserved.

`rerank.rs` exposes:

```rust
#[async_trait::async_trait]
pub trait RerankBackend: Send + Sync {
    async fn score(&self, query: &str, candidates: &[&str]) -> Result<Vec<f32>, RerankError>;
    fn is_ready(&self) -> bool; // false until model is loaded into memory
}

pub async fn apply_reranker(
    hits: Vec<RecallHit>,
    query: &str,
    backend: &dyn RerankBackend,
    timeout: Duration,
) -> Vec<RecallHit>;
```

`apply_reranker`:

1. Returns `hits` unchanged if `backend.is_ready()` is `false` (model still loading) and logs `WARN` once per process.
2. Builds `candidates: Vec<&str>` from `hits[i].summary.as_deref().unwrap_or("")`.
3. Calls `tokio::time::timeout(timeout, backend.score(query, &candidates))`.
4. On `Ok(scores)`: replaces `hits[i].score = scores[i] as f64`, sets `search_type = Hybrid` (rerank is part of hybrid), sorts descending by score, returns.
5. On `Err(_)` (timeout) or backend error: returns `hits` unchanged (RRF order preserved) and increments a metric counter `recall_reranker_fallback_total`.

The `FastembedReranker` adapter (in `recall/fastembed.rs`) holds a `OnceCell<TextRerank>` that is lazily initialized on first `score()` call. Model loading uses `fastembed::TextRerank::try_new(InitOptions { model_name: TextRerank::BGERerankerV2M3, cache_dir: model_cache_dir(), .. })`. `fastembed-rs 4.x` memory-maps the model file by default, so subsequent loads in the same process are near-instant.

### `clx model fetch` Command

```bash
clx model fetch                # blocking, with progress bar
clx model fetch --background   # spawns detached child; returns immediately
clx model fetch --force        # re-downloads even if present
```

Implementation in `crates/clx/src/commands/model.rs` (~150 LoC):

- Resolves model name and target directory under `~/.clx/models/`.
- Computes expected SHA-256 from a hardcoded constant (pinned per release).
- Spawns `fastembed::TextRerank::try_new` in a blocking thread; surfaces progress via `indicatif` if foreground, swallows output if `--background`.
- On `--force`, deletes the existing directory first.
- Exit codes: `0` success, `1` network error, `2` checksum mismatch, `3` already running (lockfile at `~/.clx/models/.fetch.lock`).
- Writes a marker `~/.clx/models/bge-reranker-v2-m3/.ready` after successful checksum verification. `FastembedReranker::is_ready()` checks for this marker.

### First-Run Hook Behavior

In `crates/clx-hook/src/hooks/user_prompt_submit.rs`:

## D3. Multiplicative Time-Decay and Percentile Gate

### Files to modify

- `crates/clx-core/src/recall/engine.rs` — chain `apply_time_decay` and `apply_percentile_gate` after the reranker.
- `crates/clx-core/src/recall/decay.rs` — new file (~120 LoC) with `apply_time_decay` and `apply_percentile_gate` as pure functions.
- `crates/clx-core/src/config.rs` — add `AutoRecallConfig::time_decay_half_life_days` (default 30) and `percentile_gate` (default 70).

### Algorithm

**Time-decay** (multiplicative half-life model):

```text
final_score = base_score * exp(-ln(2) * age_days / half_life_days)
```

At `age_days == half_life_days` the score is halved. At `age_days == 0` it is unchanged. Multiplicative ensures that even a perfect semantic match from a year ago fades below recent partial matches; this is what real retrieval-tuning research (Drift et al. 2024) recommends over additive decay.

**Percentile gate:**

After time-decay, compute the p70 of all hit scores. Drop hits below that percentile. With 10 candidates, that means keeping the top 3. With 50 candidates, the top 15. The intent is "show only the obviously good stuff" rather than the brittle fixed `0.35` threshold in 0.7.x.

**Recent-session pin (Group B's pin_recent_sessions) is injected before the gate** so pinned summaries are never dropped by percentile filtering. This means the pin module must call `apply_percentile_gate(hits_after_pin)` rather than relying on the engine alone — documented in the engine pipeline.

### Pure Function Signatures

```rust
pub fn apply_time_decay(hits: Vec<RecallHit>, now: DateTime<Utc>, half_life_days: u32) -> Vec<RecallHit>;
pub fn apply_percentile_gate(hits: Vec<RecallHit>, percentile: u8) -> Vec<RecallHit>;
```

Both are deterministic, take owned input, return owned output. `now` is injected (not `Utc::now()`) so tests are deterministic.

```yaml
      # ... plus 30 distractor snapshots shared across all queries
  # ... 49 more items
```

The fixture snapshots are appended to one shared pool (~80 to 120 distinct snapshots across the whole file), so a query for "Azure retries" must surface 12 and 17 out of the full corpus, not out of a query-specific 2-item bucket.

### Generation Methodology

`scripts/generate_golden_set.py`:

1. Reads every `specs/*.md` and produces synthetic "what did we decide about X" questions using a deterministic template against section headers.
2. Pulls the public GitHub issues list via `gh issue list --state all --json title,body --limit 200` and converts titles into "how did we handle X" questions.
3. For each generated question, deterministically picks the 1 to 3 most relevant spec sections as `expected_snapshot_ids`.
4. Writes one YAML file. The script is reproducible via the embedded `seed: 42` for any shuffling step.

All inputs are public (specs in this repo, GitHub issues on the public tracker). No private session content is read.

### Metrics

For each query:

```text
hits      = top-K results from RecallEngine
expected  = set of expected_snapshot_ids
retrieved = set of hits.snapshot_id

precision_at_k = |retrieved ∩ expected| / K
recall_at_k    = |retrieved ∩ expected| / |expected|
```

RAGAS-style aggregates across all 50 queries:

```text
context_precision = mean(precision_at_k for q in queries)
context_recall    = mean(recall_at_k for q in queries)
```

The bench prints both numbers and a one-line summary; criterion records the per-query latency separately.

### `cargo bench` Structure

```rust
// crates/clx-core/benches/recall_accuracy.rs
use criterion::{Criterion, criterion_group, criterion_main};

fn bench_recall_accuracy(c: &mut Criterion) {
    let golden = load_golden_set("tests/fixtures/recall_golden.yaml");
    let storage = seed_storage_from_fixtures(&golden);
    let engine = RecallEngine::new(&storage, None, None);

    let mut group = c.benchmark_group("recall_accuracy");
    group.sample_size(10);
    group.bench_function("synthetic_golden_50", |b| {
        b.iter(|| {
            let (precision, recall) = run_evaluation(&engine, &golden);
            assert!(precision.is_finite());
            (precision, recall)
        });
    });
    group.finish();

    let (precision, recall) = run_evaluation(&engine, &golden);
    println!("\nRAGAS metrics");
    println!("  context_precision = {precision:.3}");
    println!("  context_recall    = {recall:.3}");
    if precision < 0.85 {
        eprintln!("::warning::context_precision {precision:.3} below 0.85 threshold");
        std::process::exit(0); // warn-only in 0.8.0
    }
}

criterion_group!(benches, bench_recall_accuracy);
criterion_main!(benches);
```

### Risks and Mitigations

| Risk                                                                              | Mitigation                                                                                                          |
| --------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| Synthetic queries do not represent real user prompts                              | Documented limitation; user-derived set planned for 0.8.1 once a PHI/secrets review pipeline is in place            |
| 50 queries is statistically thin                                                  | Bench treats results as a directional signal, not a SLA; `context_precision` is a guardrail, not a contract         |
| Golden set drifts as specs evolve                                                 | Regenerate on every release; the seed and generator script make this a one-line CI job (deferred to 0.8.1)          |
| Bench flaky on slow CI runners                                                    | `sample_size(10)` for criterion; metric thresholds are static, not relative-to-runtime                              |
| Accidental commit of real session content via the generator                      | Smoke test 5 above scans the YAML; PR review checklist includes a manual diff scan                                 |
