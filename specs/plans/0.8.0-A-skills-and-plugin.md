# CLX 0.8.0 Group A: Skills + Plugin Migration to 2026 Schema

**Branch:** `feat/0.8.0-memory-skills-coverage`
**Phase:** 1 of the 0.8.0 release plan (per design spec §4)
**Estimated duration:** 1 working day (6-8 person-hours, single subagent)
**Owner-subagent:** docs-architect
**Blocking?** No (parallel-safe with Phases 2-7); however, downstream `claude /skills list` smoke depends on this phase landing first.
**Date:** 2026-05-16

---

## 1. Goal

Migrate the CLX Claude Code plugin from the legacy single-skill 2025 layout
(`plugin/plugin.json` + `plugin/skills/using-clx/SKILL.md`) to the Claude Code
2026 plugin schema, and split the monolithic `using-clx` skill into six
model-discoverable named skills.

Success means: a clean `claude` install with the CLX plugin enabled reports all
six skills via `claude /skills list`, each skill triggers on its target user
phrase, no trigger-bleed (per claude-mem issue #1651 reference in §3.5 of the
design spec), and the static `validate.sh` gate runs green in CI.

---

## 2. Scope (in / out)

In scope:

- File reorganization under `plugin/.claude-plugin/`
- New `plugin/.claude-plugin/plugin.json` per 2026 schema
- Six new `SKILL.md` files under `plugin/.claude-plugin/skills/<name>/`
- Rewritten `plugin/scripts/validate.sh` that understands the 2026 layout
- New `plugin/scripts/migrate.sh` to help existing users move local installs

  for `validate.sh`

  `plugin/.archive/2025/using-clx/SKILL.md` for traceability

- Recall pipeline, RRF, reranker
- `tool_events` table or aggregator
- Auto-summarize, pin-recent-sessions, `clx trust`
- Coverage push, mutation testing, RAGAS bench
- Brew formula bump (Phase 13)

## 3. Step-by-step Task List

### 3a. Move and rewrite plugin manifest (1 hr)

1. Create directory `plugin/.claude-plugin/`.
2. Move `plugin/plugin.json` to `plugin/.claude-plugin/plugin.json` via
   `git mv`. Preserve history.
3. Rewrite the manifest body to match the 2026 schema. New top-level keys:
   `name`, `version`, `description`, `homepage`, `skills` (array of relative
   paths), `mcp_servers` (empty for now; CLX MCP is registered out of band by
   `clx install`), `entry_point` (omitted; this plugin has no command surface).
4. Bump `version` to `0.8.0`.
5. Update `description` to a one-sentence summary that does NOT trigger-bleed
   (avoid generic phrases like "memory" without context; reference CLX
   specifically).
6. Commit: `chore(plugin): move plugin.json to .claude-plugin/ for 2026 schema`.

### 3b. Create six named skills (3 hr)

For each of the six skills, create directory
`plugin/.claude-plugin/skills/<name>/` containing one file `SKILL.md` with a
YAML frontmatter block and a markdown body. Body length targets per design
spec §3.5. Frontmatter `description` must be <= 1024 chars and start with a
user-facing trigger phrase.

Skill 1: `clx-recall`

- Frontmatter:
  - `name: clx-recall`
  - `description:` starts with "Use when user references earlier sessions,
    past decisions, or asks 'what did I do' about prior work."
  - Lists 3 example phrases: "look up our Azure discussion", "find
    checkpoints from yesterday", "what did we decide about retries".
  - Names the MCP tool: `mcp__clx__clx_recall`.
- Body outline (~80 LoC): query craft, when NOT to use (real-time data,
  current-session-only context, secrets), result interpretation (score
  bands, time-decay note), failure modes (embedding model changed, empty
  index), one worked example query plus result.
- Anti-pattern guard: explicitly state "do not invoke for current-session
  context; Claude already has it in the transcript."

Skill 2: `clx-remember`

- Frontmatter:
  - `name: clx-remember`
  - `description:` starts with "Use when the user states a durable
    preference, decision, or fact worth carrying across sessions."
  - Examples: "remember that we use Conventional Commits", "save this
    decision about Azure tenant routing", "note that prod uses M-series
    only".
  - MCP tool: `mcp__clx__clx_remember`.
- Body outline (~60 LoC): what qualifies as a durable memory (decisions,
  preferences, gotchas), what does NOT (one-off code, transient state),
  how to phrase the saved fact (declarative, dated context if relevant),
  one worked example.
- Anti-pattern guard: do not save secrets, PII, or speculative drafts;
  prefer `clx_checkpoint` for large in-flight context.

Skill 3: `clx-checkpoint`

- Frontmatter:
  - `name: clx-checkpoint`
  - `description:` starts with "Use immediately before a risky, large, or
    hard-to-reverse change so the pre-change context is recoverable."
  - Examples: "checkpoint before refactor", "snapshot the current state
    before migration", "save context before squash".
  - MCP tool: `mcp__clx__clx_checkpoint`.
- Body outline (~50 LoC): what counts as risky (mass-rename, schema migr,
  squash, force-push), how a checkpoint differs from `clx_remember`
  (full-session snapshot vs single fact), retrieval pattern (recall after
  failure).
- Anti-pattern guard: do not checkpoint on every assistant turn (that is
  what `memory.auto_summarize` is for, off by default in 0.8.0).

Skill 4: `clx-rules`

- Frontmatter:
  - `name: clx-rules`
  - `description:` starts with "Use when the user asks 'what rules apply
    here', after a long session where CLAUDE.md instructions may have
    decayed from context, or when rule violations seem likely."
  - Examples: "what coding rules apply", "refresh project rules",
    "remind me of the commit convention".
  - MCP tool: `mcp__clx__clx_rules`.
- Body outline (~70 LoC): how CLX surfaces project, user, and org rules;
  precedence (org > user > project); when to refresh proactively (after
  /compact, after 50+ turns, on a new file area), example output shape.
- Anti-pattern guard: do not invoke at the start of every turn; rules are
  already injected via CLAUDE.md. Use only on staleness signals.

Skill 5: `clx-resume`

- Frontmatter:
  - `name: clx-resume`
  - `description:` starts with "Use when the user wants to resume earlier
    work on a specific topic, ticket, or file area from a prior session."
  - Examples: "resume the recall accuracy work", "pick up where we left
    off on the Azure fallback", "continue the dashboard refactor".
  - MCP tools used: `mcp__clx__clx_recall` followed by reading retrieved
    snapshots; this skill is workflow guidance, not a single tool call.
- Body outline (~90 LoC): step-by-step resume protocol (1) recall by
  topic, (2) read most recent snapshot summary, (3) confirm with user
  before acting, (4) optional `clx_checkpoint` to mark resume point.
  Includes a worked example.
- Anti-pattern guard: do not auto-execute steps from a prior session
  without user confirmation; recalled context may be stale or superseded.

Skill 6: `clx-doctor`

- Frontmatter:
  - `name: clx-doctor`
  - `description:` starts with "Use when CLX recall returns no results,
    looks empty, or behaves unexpectedly; diagnoses provider, embedding,
    DB, and config issues."
  - Examples: "why isn't recall working", "clx returns empty", "diagnose
    embedding model mismatch", "check provider fallback status".
  - Tools used: shell `clx doctor`, `clx embeddings status`,
    `clx providers ping`, plus log inspection.
- Body outline (~120 LoC): diagnostic decision tree (empty recall ->
  check index size; embedding mismatch -> rebuild; provider 5xx ->
  inspect fallback cooldown; missing snapshots -> check Stop hook is
  firing; corrupt DB -> back up and rebuild). Each branch references a
  shell command and expected output shape.
- Anti-pattern guard: do not invoke for normal "no relevant context"
  outcomes; only for outright failures or recall results that look
  structurally wrong.

Commit each skill atomically:
`docs(plugin): add clx-recall skill (2026 schema)` and so on, one commit
per skill.

### 3c. Update `validate.sh` for 2026 schema (1 hr)

7. Rewrite `plugin/scripts/validate.sh` to:
   - Locate manifest at `plugin/.claude-plugin/plugin.json`. If the legacy
     `plugin/plugin.json` exists, fail with a message pointing at
     `migrate.sh`.
   - Iterate every directory under `plugin/.claude-plugin/skills/`,
     require each to contain exactly one `SKILL.md`.
   - For each `SKILL.md`: extract YAML frontmatter, require keys `name`
     and `description`. Require `name` matches the parent directory name
     (case-sensitive). Require `description` length, after newline
     folding, is <= 1024 chars. Require `description` is non-empty and
     does not contain a tab character (YAML safety).
   - Require the manifest `skills` array enumerates exactly the same set
     of directory names that exist on disk (no orphans, no missing).
   - Exit 0 on success with a one-line OK summary; exit non-zero with a
     specific FAIL message on any check.
8. Add a `--strict` flag (default on in CI) that also requires each
   description to start with the literal token "Use when" as a
   trigger-bleed guard.
9. Add `set -euo pipefail` (already present) and `LC_ALL=C` for portable
   sorting.
10. Commit: `chore(plugin): rewrite validate.sh for 2026 plugin schema`.

### 3d. Write `migrate.sh` for existing users (1.5 hr)

11. Create `plugin/scripts/migrate.sh`:
    - Detect legacy `plugin/plugin.json` and/or
      `plugin/skills/using-clx/SKILL.md`.
    - Print a plan of moves before doing anything; require an explicit
      `--yes` flag or interactive `y` confirmation to proceed.
    - For a user's installed plugin under
      `~/.claude/plugins/<plugin-id>/`: move `plugin.json` to
      `.claude-plugin/plugin.json` and remove the old `skills/` tree
      after archiving it to `.archive/2025/skills/` next to the new
      layout.
    - For the CLX source repo: no-op if `.claude-plugin/` already exists;
      otherwise apply the same moves.
    - Provide `--dry-run` (print, do not write) and `--rollback`
      (reverse the moves using the archive copy).
    - Exit non-zero with actionable messages on any error.
12. Commit: `chore(plugin): add migrate.sh for 2025 to 2026 plugin layout`.

### 3e. Archive legacy skill content (0.5 hr)

13. Move `plugin/skills/using-clx/SKILL.md` to
    `plugin/.archive/2025/using-clx/SKILL.md` via `git mv` so blame is
    preserved. Remove the now-empty `plugin/skills/` directory.
14. Commit: `chore(plugin): archive legacy using-clx skill under .archive/2025/`.

### 3f. Wire up the manifest (0.5 hr)

15. Update `plugin/.claude-plugin/plugin.json` so its `skills` array lists
    all six directory names, in the order: `clx-recall`, `clx-remember`,
    `clx-checkpoint`, `clx-rules`, `clx-resume`, `clx-doctor`.
16. Run `validate.sh` locally; resolve any failures.
17. Commit: `chore(plugin): register six 2026 skills in plugin.json`.

## 4. Frontmatter Contract (golden fixture)

Every `SKILL.md` must conform to this contract. The validator enforces it,
and `tests/fixtures/plugin/golden_frontmatter.yaml` carries a canonical
example for round-trip testing.

```yaml
---
name: clx-recall              # required, must match parent dir
description: >                # required, <= 1024 chars after folding
  Use when user references earlier sessions, past decisions, or asks
  "what did I do" about prior work. Invokes mcp__clx__clx_recall.
  Examples: "look up our Azure backend discussion", "find checkpoints
  from yesterday", "what did we decide about retries". Do not use for
  current-session context or real-time data.
---
```

Hard limits:

- `name` is kebab-case, 3 to 40 chars, matches `^[a-z][a-z0-9-]+$`.
- `description` <= 1024 chars after newline-to-space folding.
- `description` starts with "Use when" (strict mode).
- No tabs anywhere in the frontmatter block.
- Body starts after the closing `---` with a single H1 line matching the
  skill display name.

## 5. Test Plan

### 5.1 Shell tests on `validate.sh`

Add `plugin/scripts/tests/validate_test.sh` (bash, runnable standalone and
from CI). Cases:

1. Happy path: real `plugin/.claude-plugin/` tree validates green, exit 0.
2. Missing manifest: temporary tree without `plugin.json`, expect FAIL
   pointing at `migrate.sh`.
3. Legacy layout detected: temp tree with only legacy `plugin/plugin.json`,
   expect FAIL with migration hint.
4. Skill name mismatch: rename dir but not frontmatter `name`, expect
   FAIL on mismatch.
5. Oversize description: synthesise a 1100-char description, expect FAIL.
6. Trigger-bleed (strict mode): description does not start with "Use
   when", expect FAIL in strict, OK in non-strict.
7. Orphan in manifest: `skills` array references a non-existent dir,
   expect FAIL.
8. Orphan on disk: dir exists but is not in `skills`, expect FAIL.

Test runner: plain `bash` plus a tiny `assert_eq` shell helper; no `bats`
dependency to keep CI light.

### 5.2 Golden frontmatter fixture

`tests/fixtures/plugin/golden_frontmatter.yaml` plus a Rust unit test in
`crates/clx-core/tests/plugin_frontmatter_golden.rs` (or a pure shell
diff) that parses the fixture, asserts every required key is present, and
prints the byte length of `description`. Acts as a regression net if the
schema shifts again.

### 5.3 Manual integration smoke

After validate passes:

1. `claude plugin install ./plugin` (or symlink) on a clean profile.
2. `claude /skills list`: must show exactly six entries with the expected
   names.
3. In a test session, type one trigger phrase per skill and confirm
   Claude proposes invoking the matching skill (no trigger-bleed across
   skills). Record the transcript at
   `tests/manual/plugin_smoke_2026-05-16.md`.

### 5.4 CI wiring

Add a CI job `plugin-validate` that runs `plugin/scripts/validate.sh` and
`plugin/scripts/tests/validate_test.sh` on every PR that touches
`plugin/**`. Job is fast (<10 s) and gating.

## 7. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Claude Code 2026 schema shifts between draft and release | Low | High (re-do migration) | Pin to documented schema as of 2026-05-16; keep validator strict so future drift is caught immediately. Subscribe to plugin-schema release notes. |
| Trigger bleed across the six skills causes Claude to pick wrong one | Medium | Medium (UX regression) | Each description starts with a narrow "Use when" clause and lists 3 concrete examples; smoke test in 5.3 catches early. Strict-mode validator enforces the "Use when" prefix. |
| Existing users get stuck after upgrade (plugin not discovered) | Medium | High (silent failure) | `migrate.sh` is shipped in 0.8.0 and called out in release notes; old layout is detected by `validate.sh` with an actionable error. Brew formula post-install hint added in Phase 13. |
| `migrate.sh` corrupts a user's installed plugin | Low | High (data loss in plugin dir) | Archive-then-move pattern, `--dry-run` default for interactive flow, `--rollback` flag, mandatory `--yes` for non-interactive. No `rm -rf`. |
| Body content drifts from real CLX behavior over time | Medium | Medium (stale guidance) | Each skill body has a `_Last verified:_` footer date; CI lint warns when older than 90 days. Tracked outside this phase, ticketed for 0.8.1. |
| Validator depends on `python3` for JSON parsing (current script uses it) | Low | Low | Already a CI/dev baseline dependency; document in `plugin/README.md`. Fallback: `jq` check added in strict mode. |
| Six skills exceed Claude's "skills shown" UX cap | Low | Low | Six is well under documented 2026 limits; if cap appears, consolidate `clx-recall` and `clx-resume` (they share the recall surface). |

## 10. Sequencing Notes

Inside this phase, the optimal sub-sequence for a single subagent is:

1. Steps 3a (manifest move) and 3e (archive) first, so the tree is in a
   clean state.
2. Step 3c (rewrite validator) next, before authoring skills, so each
   skill can be validated as it lands.
3. Steps 3b skills in parallel-friendly order (`clx-recall` first since
   the others reference its patterns).
4. Step 3d (migrate.sh) after the layout is final.
5. Step 3f (wire manifest) last.

If parallelizing across two subagents:

- Subagent A: 3a, 3c, 3d, 3f, tests.
- Subagent B: 3b (all six skill bodies), 3e archive.

File ownership is disjoint, so no merge conflicts.
