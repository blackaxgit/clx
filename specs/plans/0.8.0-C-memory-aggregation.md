## 0. Group Overview

Group C delivers the three "memory aggregation" features that compose CLX's signal pipeline:

1. **C1 `tool_events`** — Per-tool aggregated signal with 60s windowed dedup (additive, always-on for mutators).
2. **C2 Pin recent sessions** — Opt-in injection of the last N session summaries into every `UserPromptSubmit` recall block.
3. **C3 Auto-summarize** — Opt-in rolling N-turn summarization, fires on `Stop`, produces `AutoSummary`-triggered snapshots.

**Layering invariants (per CLAUDE.md):**

- Domain / pure logic lives in `clx-core` (`storage/tool_events.rs`, `summarize.rs`).
- Orchestration / hook wiring lives in `clx-hook` (`hooks/post_tool_use.rs`, `hooks/subagent.rs`, new `hooks/stop_auto_summary.rs`).
- All three sub-phases must leave `cargo clippy --workspace -- -D warnings` clean and add tests that exercise both happy and failure paths.

**Migration note (important):** The current repository has no `migrations/` directory. All schema migrations live inline in `crates/clx-core/src/storage/migration.rs` as `migrate_to_vN()` methods, gated by `SCHEMA_VERSION` (currently 5). The design spec's reference to `migrations/0008_tool_events.sql` is therefore a logical name; the actual implementation is a new `migrate_to_v6()` method. The plan reflects this reality and keeps the SQL DDL fenced inside the Rust migration so all schema lives in one auditable place.

**Total estimated effort:** ~4.5 person-days (C1: 2d, C2: 0.5d, C3: 2d).

## C1 — `tool_events` Table + Aggregator (Phase 4, ~2d)

### C1.1 Scope and contract

Aggregate every mutating tool call into a deduplicated `tool_events` row that recall can join against later. Pure additive change: existing `events` append in `post_tool_use.rs:47` is untouched; the new aggregator runs **after** the event append (line 49) and only fires for whitelisted mutator tools.

**Mutator whitelist (exact match on `tool_name`):**

```rust
pub(crate) const MUTATOR_TOOLS: &[&str] = &["Edit", "Write", "MultiEdit", "NotebookEdit"];
```

Plus `Bash` when the command matches a mutator pattern (regex over the `command` field):

```
^\s*(git\s+(commit|push|reset|rebase|merge|cherry-pick|checkout\s+-{1,2})|rm\s|cargo\s+install|npm\s+install|pip\s+install|mv\s|cp\s|chmod\s|chown\s|>\s*/)
```

Read-only tools (`Read`, `Grep`, `Glob`, `WebFetch`, `LS`, `mcp__clx__clx_recall`, etc.) are silently ignored.

### C1.3 Schema (embedded in `migrate_to_v6`)

```sql
CREATE TABLE IF NOT EXISTS tool_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT NOT NULL,
  tool_name TEXT NOT NULL,
  target TEXT,
  summary TEXT NOT NULL,
  outcome TEXT NOT NULL,
  window_start_unix INTEGER NOT NULL,
  window_end_unix INTEGER NOT NULL,
  occurrence_count INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (session_id) REFERENCES sessions(id)
);
CREATE INDEX IF NOT EXISTS idx_tool_events_session
  ON tool_events(session_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_tool_events_target
  ON tool_events(target);
CREATE INDEX IF NOT EXISTS idx_tool_events_dedup
  ON tool_events(tool_name, target, window_end_unix);
```

The migration must also `INSERT OR IGNORE` the `sessions` row before any insert in the aggregator (FK regression pattern proven in `audit.rs:22-26` — see `crates/clx-core/src/storage/audit.rs`).

### C1.4 60-second dedup logic

```text
on PostToolUse(tool, input, outcome):
  if tool not in MUTATOR_TOOLS:
    if tool == "Bash" and is_mutator_bash(input.command):
      proceed
    else:
      return
  target  = derive_target(tool, input)        // normalized path or command verb
  summary = derive_summary(tool, input, outcome)
  now     = unix_now()
  cutoff  = now - 60

  // INSERT OR IGNORE sessions(session_id) first
  // Then within a transaction:
  row = SELECT id FROM tool_events
        WHERE tool_name = ? AND COALESCE(target,'') = COALESCE(?,'')
          AND window_end_unix >= ?
        ORDER BY id DESC LIMIT 1
  if row:
    UPDATE tool_events
      SET occurrence_count = occurrence_count + 1,
          window_end_unix  = ?,
          outcome          = ?,   // keep latest
          summary          = ?    // keep latest line
      WHERE id = ?
  else:
    INSERT INTO tool_events (...) VALUES (...)
```

Target normalization rules (in `aggregator::derive_target`):

- `Edit` / `Write` / `MultiEdit` / `NotebookEdit`: canonicalize `file_path` via `std::path::Path::components` (strip `./`, collapse `..`, no symlink resolution to keep pure).
- `Bash`: lowercase the first verb token (e.g., `git commit -m ...` -> `git-commit`).
- Missing path/command: `None`. `COALESCE(...,'')` handles `NULL` in dedup.

### C1.5 Deterministic summary templates

| Tool | Template | Source fields |
|------|----------|---------------|
| `Edit` | `edit <basename> (chars: <old_len>->\<new_len>)` | `old_string`, `new_string` |
| `Write` | `write <basename> (<bytes> bytes)` | `content.len()` |
| `MultiEdit` | `multi-edit <basename> (<n> edits)` | `edits.len()` |
| `NotebookEdit` | `notebook <basename> cell <cell_id_or_idx>` | `notebook_path`, `cell_id` |
| `Bash` | `bash <verb>: <truncated 80 chars>` | first verb, sanitized truncated command |

All templates are deterministic and stable. Truncation uses `chars().take(80).collect()` (not bytes) to keep UTF-8 safe.

### C2.4 Storage query

```sql
SELECT s.session_id, MAX(s.created_at) AS created_at, s.summary
FROM snapshots s
WHERE s.summary IS NOT NULL
  AND s.session_id != ?1
GROUP BY s.session_id
ORDER BY MAX(s.created_at) DESC
LIMIT ?2;
```

The `GROUP BY session_id` keeps the latest snapshot per session. Self-pin guard is the `session_id != ?1` predicate, not a post-filter, so we honor `LIMIT N` correctly.

## C3 — Auto-Summarize Mode (Phase 10, ~2d)

### C3.1 Scope and contract

Opt-in. When `memory.auto_summarize.enabled = true`, a new `Stop` hook handler counts assistant turns since the last `AutoSummary` snapshot for the session. When `turns_since >= every_n_turns` and `skip_when_idle` allows it, the handler:

1. Reads the last N turns from the transcript file (`transcript_path` in hook envelope).
2. Calls the configured summarizer via `Config::create_llm_client(Capability::Chat)`.
3. Issues one chat completion with a deterministic template.
4. Writes the result as a `Snapshot` with `trigger = SnapshotTrigger::AutoSummary`.

If the LLM call fails (no key, network, timeout, missing capability route), falls back to a deterministic template-based summarizer (no LLM) that concatenates "files touched" + "last user request" — never panics, never blocks.

### C3.5 Storage helper

```rust
pub fn turns_since_last_auto_summary(&self, session_id: &str) -> Result<usize> {
    // 1. Find last auto_summary snapshot timestamp (or session start if none).
    // 2. Count events of type 'message' where role='assistant' since that timestamp.
    //    NOTE: current events table has no role column; we count event_type='message'
    //    AND tool_name IS NULL as a heuristic for assistant turns. Documented in
    //    a `// HEURISTIC:` comment; precise turn counting deferred to a future
    //    transcript-parsing helper if needed.
    let last_ts: Option<String> = self.conn.query_row(
        "SELECT MAX(created_at) FROM snapshots
         WHERE session_id = ?1 AND trigger = 'auto_summary'",
        [session_id],
        |row| row.get(0),
    ).optional()?;

    let count: i64 = match last_ts {
        Some(ts) => self.conn.query_row(
            "SELECT COUNT(*) FROM events
             WHERE session_id = ?1
               AND event_type = 'message'
               AND timestamp > ?2",
            params![session_id, ts],
            |row| row.get(0),
        )?,
        None => self.conn.query_row(
            "SELECT COUNT(*) FROM events
             WHERE session_id = ?1 AND event_type = 'message'",
            [session_id],
            |row| row.get(0),
        )?,
    };
    Ok(count as usize)
}
```

```rust
    turns: &[TurnSlice<'_>],
    cfg: &AutoSummarizeConfig,
    llm: Option<&dyn LlmBackend>,
) -> Result<String> {
    if turns.is_empty() {
        return Ok(String::new());
    }
    if let Some(client) = llm {
        match try_llm_summary(client, turns, cfg).await {
            Ok(s) if !s.trim().is_empty() => return Ok(truncate_chars(s, cfg.max_summary_chars)),
            Ok(_) | Err(_) => { /* fall through to template */ }
        }
    }
    Ok(template_fallback(turns, cfg.max_summary_chars))
}
```

- `build_prompt`: `"Summarize the following N-turn span into <= {max_summary_chars} chars, focusing on decisions made, files touched, and TODOs.\n\n{turns}"`.
- `template_fallback`: deterministic, no LLM. Extracts last user request (first 200 chars), counts assistant turns, lists distinct file paths mentioned via a simple regex `[A-Za-z0-9_./-]+\.(rs|toml|md|sql|yaml|py|ts|tsx|js)`. Formats as `Auto-summary (N turns, no LLM): <user_req> | files: a.rs, b.toml`.
- Pure: depends only on `LlmBackend` trait, `AutoSummarizeConfig`, and standard regex; no `Config::load()`, no `Storage`, no filesystem.

### C3.7 Hook handler (`stop_auto_summary.rs`)

```rust
pub(crate) async fn handle_stop_auto_summary(input: HookInput) -> Result<()> {
    let config = Config::load().unwrap_or_default();
    if !config.memory.auto_summarize.enabled {
        return Ok(());
    }

    let storage = match Storage::open_default() { Ok(s) => s, Err(e) => { warn!(...); return Ok(()); } };
    let turns_since = storage.turns_since_last_auto_summary(&input.session_id).unwrap_or(0);
    let cfg = &config.memory.auto_summarize;

    if (turns_since as u32) < cfg.every_n_turns {
        debug!("auto-summary: {turns_since} < {} turns, skipping", cfg.every_n_turns);
        return Ok(());
    }

    if cfg.skip_when_idle && !had_mutator_activity(&storage, &input.session_id, turns_since)? {
        debug!("auto-summary: skip_when_idle and no mutator activity since last summary");
        return Ok(());
    }

    let transcript = match input.transcript_path.as_ref() {
        Some(p) => parse_transcript_tail(p, turns_since).unwrap_or_default(),
        None => Vec::new(),
    };
    if transcript.is_empty() {
        return Ok(());
    }

    let llm = config.create_llm_client(parse_capability(&cfg.summarizer_capability)).ok();
    let llm_ref = llm.as_deref();
    let summary = clx_core::summarize::summarize_turns(&transcript, cfg, llm_ref).await
        .unwrap_or_else(|_| String::new());
    if summary.is_empty() { return Ok(()); }

    let mut snap = Snapshot::new(input.session_id.clone(), SnapshotTrigger::AutoSummary);
    snap.summary = Some(summary);
    snap.message_count = Some(turns_since as i32);
    if let Err(e) = storage.create_snapshot(&snap) {
        warn!("auto-summary: failed to persist snapshot: {e}");
    }
    Ok(())
}
```

Helpers:

- `had_mutator_activity(storage, session_id, n) -> Result<bool>` — queries `tool_events` (from C1!) for any row with `created_at > last_auto_summary_timestamp`. If C1's table exists and has a row, return `true`. C2/C3 thereby depend on C1 being merged first.
- `parse_transcript_tail(path, n) -> Result<Vec<TurnSlice>>` — reuses existing transcript-reading helpers in `clx-hook/src/transcript.rs` (already used by `post_tool_use.rs::count_transcript_tokens`); add a new `last_n_turns(path, n) -> Result<Vec<TurnSlice>>` parallel function.
- `parse_capability(s) -> Capability` — small match on the string ("chat", "embeddings", ...) with a fallback to `Capability::Chat`.

### C3.11 Risks

| Risk | Mitigation |
|------|------------|
| Heuristic "assistant turn = message event with NULL tool_name" undercounts true turns | Documented as best-effort; over- or under-counting only changes *when* summary fires, not correctness; precise transcript parsing deferred. |
| LLM call exceeds Stop hook latency budget | `summarize_turns` is `async`; wrap in `tokio::time::timeout(cfg.timeout_ms or default 5s)`; on timeout fall through to template. |
| Recursive auto-summary loops (summary itself creates a message event) | `turns_since` counts events strictly **after** the latest auto_summary snapshot timestamp; snapshot creation is the demarcation. |
| Mutator-activity guard depends on C1 | Phase order: C1 lands before C3 (already reflected in §4 of design spec, phases 4 -> 10). If C1 ships disabled, `skip_when_idle` defaults to false branch ("no tool_events table" = treat as idle? no — graceful fallback: if `tool_events` query errors, assume **not** idle and proceed). |
| Capability route missing in config | `parse_capability` defaults to `Capability::Chat`; `create_llm_client` Err -> `llm = None` -> template fallback. |
| `every_n_turns: 0` triggers infinite loop | Validate at config load: `assert!(every_n_turns >= 1)` or clamp; reject with warning at startup. |
