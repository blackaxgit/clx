## Executive summary

The v0.9.0 toggle is mostly well-scoped but ships with **four design-level gaps** and **one likely behavioural bug** that a hurried multi-agent pass missed. None are show-stoppers for v0.9.0 if they are documented, but several invalidate audit-chain claims if read literally. The most damaging are:

1. **PolicyEngine still loads learned rules even when L0 is disabled** — this is harmless today (no path consults the engine), but the symmetry of the L0-disabled comment ("skip deterministic ruleset entirely") is *false*; if any future code path reads `policy_engine` after the L0 gate, the L0-disable promise breaks silently. (Target 1.)
2. **SECURITY-CFG and SECURITY-ENV rows live in the *same* `audit_log` SQLite table as decision rows**, are subject to the same same-uid filesystem trust model, and **carry no signature** other than the embedded `event_fingerprint` hex string in the `reasoning` TEXT column. A same-uid attacker can `DELETE FROM audit_log WHERE layer IN ('SECURITY-CFG','SECURITY-ENV');` or `INSERT` a forged row trivially. The `tracing::warn!` anchor exists, but its anchor target is not enforced or required, and `clx-hook` is a short-lived process — there is no daemon-side `verify_fingerprint_sequence` over rows. (Target 2.)
3. **Both-off (`enabled=true, layer0_enabled=false, layer1_enabled=false`) is observationally indistinguishable from validator-enabled in the audit row stream except by the L0-DISABLED/L1-DISABLED reasoning strings.** A forensic operator who filters by `decision='blocked'` will see *nothing*; the audit DB *looks populated* (lots of L0/prompted rows) but no command was ever denied. This is by design but it is a **legitimate attack surface** for a hostile contributor who wants logs to look healthy. (Target 7.)
4. **Test 4 (`env_override_layer0_disabled_emits_security_env_audit_row`) silently double-audits when CLX_VALIDATOR_LAYER0_ENABLED=false AND the resolved config has `layer0_enabled=false`** — the env override flips the in-memory bool to false, so BOTH `security_env_overrides_active()` (SECURITY-ENV row) AND the `!config.validator.layer0_enabled` check (SECURITY-CFG row) fire on the same hook invocation. Two rows, two fingerprints, one event. The CHANGELOG/comments imply mutually exclusive sources. (Target 6.)
5. **The `audit_rows` helper in the new `pre_tool_use_l0_e2e.rs` does NOT filter SECURITY-CFG/SECURITY-ENV** (lines 106-113) — it includes them — whereas the renamed `validation_e2e.rs::audit_rows` *does* filter them. This is fine but it means Test 2's `cmd_rows` filter (`r.command == ASK_CMD`) is what excludes the SECURITY-CFG row (which has `command="<cfg-layer-disable>"`). Brittle to changes. (Target 9.)

Read the rest of this document for the file:line evidence and falsifying-test sketches.

## Target rankings (Top 10, by attacker leverage × likelihood-of-builder-error)

### T1 (SEV-HIGH-CONFIDENCE-MEDIUM): SECURITY-CFG / SECURITY-ENV rows are *not* tamper-evident in the SQLite store, only in the external sink

**File:line:** `crates/clx-hook/src/hooks/pre_tool_use.rs:118-126` (SECURITY-CFG write site), `:72-81` (SECURITY-ENV write site), `crates/clx-core/src/storage/migration.rs:188-200` (table DDL — no integrity column, no foreign-key constraint on `layer`).

**Attacker model:** same-uid local attacker (the threat model this repo already accepts everywhere else: trust-token, snapshots, age-credentials all assume same-uid is trusted-for-confidentiality, untrusted-for-integrity-evidence). Attacker wants to:
  (a) **forge** a SECURITY-CFG row to imply layers are disabled (red-herring for forensic operator), or
  (b) **delete** the SECURITY-CFG row to hide that they ran `validator.layer0_enabled: false`.

**Why the v0.9.0 implementation might be wrong:** the v0.8.2 audit_chain property in `crates/clx-hook/src/audit_chain.rs:36-43` ("tamper-evident, not tamper-proof: a local same-uid attacker with write access can forge any single-record fingerprint. The `tracing::warn!` anchor in an external sink is the reference that makes forgery detectable") was honestly written for the SECURITY-ENV row. The v0.9.0 PR extends the *signal* to a new SECURITY-CFG layer **without changing the threat model**, but the CHANGELOG (line 31-37) reads "tamper-evident audit-chain fingerprint" as if it were a primitive guarantee. It is not. The `event_fingerprint` is stored *inside* the `reasoning` TEXT column of `audit_log` (line 71 of pre_tool_use.rs `format!("...event_fingerprint={}", record.entry_hash)`), so a same-uid attacker who can `DELETE` the row erases both the row *and* its fingerprint. The `tracing::warn!` at line 106-111 is the only external anchor, and:
  - On most installs this writes to stderr, which is captured by the calling Claude Code process — same uid, same trust boundary.
  - There is no `clx doctor`-style verifier that cross-references rows-in-DB against captured warn lines.
  - The CHANGELOG `[Unreleased]` claim "External log aggregator capturing the `tracing::warn!` anchor can independently re-verify any specific disable event" is true *only if* such an aggregator is configured — and CLX ships no recommendation, no example syslog config, no aggregator wiring.

**Same-uid forgery vector:** `sqlite3 ~/.clx/data/clx.db "INSERT INTO audit_log(session_id, timestamp, command, working_dir, layer, decision, risk_score, reasoning) VALUES ('attacker-session', datetime('now'), '<cfg-layer-disable>', '/tmp', 'SECURITY-CFG', 'prompted', NULL, 'config-driven layer-disable: validator.layer0_enabled=false; event_fingerprint=$(echo -n synthetic|sha256sum|head -c64)');"`. The forged `event_fingerprint` is a 64-hex string that **passes a naive `verify_fingerprint_sequence` because no one ever calls it on rows from disk** — the function in `audit_chain.rs:157-176` is only used in unit tests against `AuditChainRecord` structs, never against `AuditLogEntry` rows.

**Same-uid deletion vector:** `sqlite3 ~/.clx/data/clx.db "DELETE FROM audit_log WHERE layer='SECURITY-CFG';"`. There is **no** WAL-replay verifier, no canary row, no hash chain anchored at session start, no Merkle root.

**Falsifying test sketch:**
1. `sqlite3` insert a synthetic SECURITY-CFG row with a hand-crafted `event_fingerprint` of all 'a' chars.
2. Run `clx audit-verify` (does this command exist? If not — that's the finding) and assert it flags the forgery.
3. Alternative: assert that `verify_fingerprint_sequence` is called somewhere on rows loaded from the DB. Grep proves it is not: `grep -rn "verify_fingerprint_sequence" crates/` returns matches only inside `audit_chain.rs` test cases.

**Falsifying property:** there exists no production code path that, given an `audit_log` row with `layer='SECURITY-CFG'`, can detect tampering of its `reasoning` column. The CHANGELOG claim must be downgraded to "tamper-evident **only against an external aggregator that the operator must set up themselves**."

### T2 (SEV-HIGH-CONFIDENCE-HIGH): PolicyEngine learned-rule load happens *before* the L0 gate, weakening the disable promise

**File:line:** `crates/clx-hook/src/hooks/pre_tool_use.rs:262-270` (engine init + `load_learned_rules`), `:275-365` (the L0 gate).

**Attacker model:** user (or attacker who can write to `~/.clx/data/clx.db`) wants the L0 disable to be a *real* disable, not "L0 rules-don't-fire-but-engine-still-loads". The comment at line 332 says "L0 disabled, skipping deterministic ruleset for '{}'" but the engine is already initialised at line 263 and learned rules already loaded at line 267.

**Why the v0.9.0 implementation might be wrong:**
  - **Functionally:** `policy_engine` is never read after line 276 when `layer0_enabled=false`. The else-branch at 328-365 does NOT call `policy_engine.evaluate(...)`. So the load is wasted work but not a security leak today.
  - **Behaviourally:** `load_learned_rules` calls `Storage::open_default()` (DB open), which has side effects on macOS keychain if `Storage::open_default()` ever touches the credentials backend (it doesn't today; verified `crates/clx-core/src/storage/migration.rs`). But it does open the SQLite DB, run migrations if needed, and read the `learned_rules` table. A user expecting "L0 disabled = no rules code runs" gets surprised.
  - **Maintenance hazard:** if any future contributor adds a "consult learned-rule whitelist before L1" hint between line 365 and the L1 gate at 399, the disable is silently bypassed.

**Falsifying test sketch:**
1. Set `validator.layer0_enabled: false`. Pre-seed `learned_rules` with a Deny pattern for `rm -rf /`.
2. Run hook with command `rm -rf /` and L1 returning `allow`.
3. Assert output is `allow`. Today this passes (L0 disabled → engine.evaluate never called → learned-rule deny is bypassed → L1 allow wins).
4. Now add a falsifying RED: assert that `learned_rules` are **never read** when `layer0_enabled=false`. This will FAIL today because line 267 reads them unconditionally.
5. The fix is to move `load_learned_rules` *inside* `if config.validator.layer0_enabled { ... }`.

**Falsifying counterexample:** a benchmark of cold-start hook invocations with a large `learned_rules` table (10k rows) under `layer0_enabled: false` shows non-zero `read_learned_rules` time. Today: yes, that time is non-zero. Fix expected: zero.

### T3 (SEV-MEDIUM-CONFIDENCE-HIGH): Double-audit when env AND config both disable L0

**File:line:** `pre_tool_use.rs:37-82` (SECURITY-ENV path runs unconditionally if env weakens) AND `:93-128` (SECURITY-CFG path runs if `config.validator.layer0_enabled` is `false` after env-override application).

**Attacker model:** not actually an attacker — a forensic operator who wants to count *events* (real layer-disable events, not log noise). The implementation duplicates rows.

**Walk-through:**
  1. `Config::load()` reads YAML where `validator.layer0_enabled: true`.
  2. `apply_env_overrides()` runs, sees `CLX_VALIDATOR_LAYER0_ENABLED=false`, sets `self.validator.layer0_enabled = false` (mod.rs:1313-1330).
  3. Hook handler `security_env_overrides_active()` at mod.rs:1247-1252 returns `[("CLX_VALIDATOR_LAYER0_ENABLED", "false")]`.
  4. pre_tool_use.rs:38 reads that → emits SECURITY-ENV row.
  5. pre_tool_use.rs:95 reads `config.validator.layer0_enabled` (now false from step 2) → emits SECURITY-CFG row too.

**Why this might be a bug:**
  - The CHANGELOG distinguishes "env-driven" vs "config-driven" semantically, but the *audit-trail* makes them join when env is the sole source.
  - The two rows have **different `event_fingerprint`s** (different `trigger_keys` strings: `"CLX_VALIDATOR_LAYER0_ENABLED"` vs `"validator.layer0_enabled=false"`) so an aggregator deduping by fingerprint sees two events. An aggregator deduping by `event_type` ("validator_disabled") still sees two records.

**Is it intentional?** The plan-spec is excluded from this recon. The CHANGELOG does not mention it. The new e2e Test 4 `env_override_layer0_disabled_emits_security_env_audit_row` (line 358-401) sets `layer0_enabled: true` in YAML and `CLX_VALIDATOR_LAYER0_ENABLED=false` env, then asserts only that *a* SECURITY-ENV row exists — it does NOT assert the *absence* of a SECURITY-CFG row. So this code path is **untested in both directions**.

**Falsifying test sketch:**
```rust
#[tokio::test]
async fn env_overrides_config_must_not_double_audit() {
    let cfg = cfg_pointing_at(&server.uri(), "  layer0_enabled: true\n  layer1_enabled: true\n");
    let env = pre_tool_use("e2e-env-double", ASK_CMD);
    let (_out, home) = run_with_extra_env(&cfg, &env, &[("CLX_VALIDATOR_LAYER0_ENABLED", "false")]);
    let rows = audit_rows(home.path(), "e2e-env-double");
    let env_rows = rows.iter().filter(|r| r.layer == "SECURITY-ENV").count();
    let cfg_rows = rows.iter().filter(|r| r.layer == "SECURITY-CFG").count();
    // Today: env_rows == 1 AND cfg_rows == 1 (double-audit).
    // Either decision is defensible, but the contract must be explicit:
    assert_eq!(env_rows + cfg_rows, 1, "exactly one audit-chain signal per disable event");
}
```

**Open risk (not in the original prompt list):** the forced-ask audit row is logged with `layer="L0"` and `reasoning="L1-DISABLED"` (lines 405-409). An operator who filters `audit_log WHERE layer='L0' AND decision='prompted'` to find "unknown-but-not-blocked" commands sees a mix of:

### T7 (SEV-MEDIUM-CONFIDENCE-HIGH): Both-off + enabled=true vs enabled=false — the "log noise theatre" attack

**File:line:** `pre_tool_use.rs:170-173` (the `validator.enabled` short-circuit), `:275-365` (L0 disabled branch which still writes an `L0-DISABLED` row).

**Attacker scenario:** a hostile contributor wants:
  - audit log to *look populated* (many rows per session) so a casual reviewer thinks validation is active,
  - but no command should ever be denied.

**Two configurations achieve this:**
  1. `validator.enabled: false` (mod.rs:580) — line 170 short-circuits *before* any audit row is written. **An operator filtering by `WHERE session_id=X AND layer LIKE 'L%'` sees zero rows.** This is loud.
  2. `validator.enabled: true, layer0_enabled: false, layer1_enabled: false` — every command writes:
      - `SECURITY-CFG` row (once per process invocation)
      - `L0` row with `reasoning='L0-DISABLED'` (every command)
      - `L0` row with `reasoning='L1-DISABLED'` (every command)
    **The audit DB looks healthy: 2-3 rows per command.** This is **silent**.

**Forensic differentiator a SOC operator must look for:** the `decision` column never equals `'blocked'` across the entire session. But this is a *negative* property — easy to miss in dashboard review. There is **no positive indicator** ("this session had zero real validation") surfaced in the dashboard.

**Why the v0.9.0 implementation might be wrong:**
The both-off configuration emits *exactly* one SECURITY-CFG row per *process invocation* (the per-hook-process scope). Across a long session with 100 commands, this is 100 SECURITY-CFG rows (one per command) + 200 L0 disabled-marker rows. A SOC operator looking at "alarm count" sees 300 alarms — high noise, easy to ignore as a misconfigured client.

**Falsifying test sketch:**
```rust
#[tokio::test]
async fn both_off_is_distinguishable_from_validator_off() {
    // Run 10 commands through both configs.
    // Assert: validator=false produces 0 audit rows.
    // Assert: both-off produces ≥ 10 SECURITY-CFG rows (one per command).
    // Assert: there exists a CLI or dashboard widget that flags both-off as
    //         "no real validation, despite enabled=true". (Today: false.)
}
```

**RED action:** add a `clx doctor` check that reads `validator.{enabled, layer0_enabled, layer1_enabled}` and emits a WARN if `enabled && !layer0_enabled && !layer1_enabled`.

### T10 (SEV-LOW-CONFIDENCE-HIGH): L0-DISABLED row carries `command` value, leaking command text into per-process audit even before read-only fast-lane

**File:line:** `pre_tool_use.rs:335-343` writes the L0-DISABLED row with `command` *as passed*. The read-only fast-lane check is at 346-362 (after the row write). For a read-only command with L0 off, the audit DB therefore gets **two** rows: `L0/L0-DISABLED/{cmd}` and `L0-READ/allowed/{cmd}`. The command appears twice.

**Attacker scenario:** none direct, but command text duplication inflates DB size and surface area for the redaction pipeline. `redact_secrets` is applied (audit.rs:33) so the secret-redaction property holds.

**Why this might be wrong:** symmetric L0/L1 design would expect *one* audit row per command per layer. Today L0-disabled writes one row regardless of outcome; if the command is read-only, a second L0-READ row also fires. Acceptable but worth noting in the CHANGELOG.

## Cross-cutting observations

### O1: Audit-log table has no `event_type` column, only `layer`
The SECURITY-CFG and SECURITY-ENV rows live in the same table as decision rows, distinguished only by the `layer` TEXT column. There is no schema-level guarantee that `layer='SECURITY-CFG'` rows are immutable or signed differently. A future schema migration that adds a CHECK constraint on `layer IN ('L0','L1','L0-READ','L1-CACHE','TRUST','SECURITY-CFG','SECURITY-ENV')` would be cheap defence-in-depth.

### O2: The `event_fingerprint` SHA-256 in the `reasoning` column is a string, not a column
This means:
  - Can't index on it.
  - Can't constrain its format.
  - A `verify_fingerprint_sequence`-style validator would need to **parse the reasoning text** to extract the fingerprint, which is brittle.
  - A first-class column (e.g. `event_fingerprint TEXT, prev_fingerprint TEXT`) would enable proper verification queries.

### O3: `tracing::warn!` is the *only* anchor
No `~/.clx/security-events.log` append-only file. No systemd journal hook. No syslog forward. If `stderr` is captured-and-discarded by the calling Claude Code wrapper, the anchor is lost. This is the **single weakest link** in the entire v0.9.0 security claim.

### O4: SECURITY-CFG/-ENV write site has no session foreign-key concern
`create_audit_log` at `storage/audit.rs:19-44` issues `INSERT OR IGNORE INTO sessions` first, so synthetic `<cfg-layer-disable>` rows succeed. Good — but a forensic operator searching by session has no way to know that these are "out-of-band" events; the FK relation looks normal.

### O5: The new e2e test file does not assert `assert_home_size_bounded` after each test
Actually it does: line 82. But each test runs in its own temp dir and the bound is the standard one. No leakage observed.
