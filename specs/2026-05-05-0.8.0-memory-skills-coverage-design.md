## 1. Scope and Goals

CLX 0.8.0 is the **memory and quality** release. It targets five user-visible outcomes and one engineering outcome:

| # | Outcome | User-facing? | Success metric |
|---|---------|--------------|----------------|
| 1 | Auto-recall returns relevant context >= 95% of the time | Yes | RAGAS context-precision >= 0.85 on synthetic golden set |
| 2 | Claude Code reliably discovers CLX skills | Yes | All 6 skills load from `.claude-plugin/` per 2026 schema |
| 3 | Pinned recent sessions show up in every prompt (opt-in) | Yes | `pin_recent_sessions: N` injects last-N summaries |

| 6 | Test coverage 72.66% -> 94-95% on instrumented denominator | No (engineering) | `cargo llvm-cov --ignore-filename-regex` reports >= 94%, warn-only gate |

**Non-goals (deferred to 0.9.0):**

- Cross-machine sync of pinned sessions
- HyDE / query rewriting (research rejected for current corpus size)
- Reranker fine-tuning (use stock `bge-reranker-v2-m3`)
- Hard-fail coverage CI gate (warn-only in 0.8.0, hard in 0.8.1)
- Real-session golden set (synthetic only in 0.8.0)

## 3. Feature Designs

### 3.1 Better Auto-Recall (target: 95% accuracy)

**Problem:** Current `hybrid_merge` (`recall.rs:346-379`) does linear weighted sum (0.6 semantic + 0.4 FTS) and dedups by `snapshot_id`. This is naive: it cannot handle rank disagreement well and has no second-stage re-scoring.

**Solution: 4-stage retrieval pipeline.**

| Stage | Tool | Output | Latency budget |
|-------|------|--------|----------------|
| 1. Candidate generation | parallel: embeddings top-50 + FTS5 top-50 | up to 100 candidates | 200 ms |
| 2. Rank fusion | Reciprocal Rank Fusion (RRF), `k = 60` | top-50 fused | 5 ms |
| 3. Cross-encoder rerank | `bge-reranker-v2-m3` via `fastembed-rs` | top-10 reranked | 250 ms |
| 4. Time-decay + gate | multiplicative `exp(-ln(2) * age_days / 30)` + p70 score gate | <= 10 final hits | 1 ms |

**Total budget: 500 ms** (same as 0.7.x, but with reranker). Falls back to linear merge on reranker failure.

**RRF formula:**

```
score(d) = sum over each ranker r of: 1 / (k + rank_r(d))
```

With `k = 60` (literature standard). RRF is parameter-free relative to score scales — it only depends on rank, so semantic and FTS disagreement is handled gracefully.

**Why bge-reranker-v2-m3:**

- Rust-native via `fastembed-rs 4.x` (no Python sidecar)
- 568 MB model, runs CPU-only on M-series (~80 ms / batch-32)
- MIT-licensed, multilingual (covers user's RU/EN content)
- No external API call (works offline; complements Azure backend gracefully)

**Time-decay rationale:**

- Multiplicative (not additive) ensures even high-similarity ancient context fades
- 30-day half-life chosen as conservative midpoint: dev sessions typically reference work from the last week, but architecture decisions stay relevant for months
- Configurable as `auto_recall.time_decay_half_life_days`

**Percentile gate:** keep only hits with `final_score >= p70(all_final_scores)`. Replaces fixed 0.35 threshold which is brittle across corpus growth.

**Recent-session pin (3.3) injected BEFORE the gate** so pinned summaries always appear.

**Insertion points (from codebase map):**

- New file `crates/clx-core/src/recall/rerank.rs` — `apply_reranker(hits, query) -> Vec<RecallHit>`
- Modify `recall.rs:159` (after `hybrid_merge` call): chain reranker -> time-decay -> gate
- Modify `recall.rs:346-379` (`hybrid_merge`) to use RRF
- Add `auto_recall.{rrf_k, reranker_enabled, time_decay_half_life_days, percentile_gate}` to config

**Evaluation: RAGAS golden set.**

- 50 hand-curated `(prompt, expected_snapshot_ids)` pairs derived from real CLX session history (after PHI/secret review)
- Stored at `tests/fixtures/recall_golden.yaml`
- New `cargo bench --bench recall_accuracy` runs RAGAS-style context-precision / context-recall
- CI gate: regression triggers if `context_precision < 0.85`

**Default: `reranker_enabled: true` in 0.8.0** (user decision 2026-05-16).
Ships with `clx model fetch` first-run UX so the 568 MB download is visible and resumable, not a silent stall.

**Risks:**

| Risk | Mitigation |
|------|------------|
| Reranker 568 MB model download blocks first recall | Eager prefetch on first `clx` invocation post-upgrade via background `clx model fetch --background`; recall falls back to RRF-only with WARN until model is ready |
| RRF + reranker latency exceeds 500 ms | Bypass reranker if `auto_recall.timeout_ms` < 400; expose `reranker_timeout_ms` (default 250 ms); per-query graceful degradation to RRF-only |
| Cold-load on M-series CPU | Memory-map model file via `fastembed-rs`; warm cache on `clx-hook` daemon start (if daemon mode enabled later) |
| Golden set leaks user content | Synthetic only in 0.8.0 (from public CLX issue tracker and specs); user-derived set deferred to 0.8.1 |

### 3.2 Explicit Save vs Auto-Save: Decision

**Research finding:** claude-mem auto-saves every assistant turn into mem://. CLX currently only saves on Stop / SubagentStop / Checkpoint. Comparison:

| Aspect | claude-mem (auto-save-all) | CLX (explicit + Stop) |
|--------|---------------------------|----------------------|
| Recall surface | Every turn | Decision points |
| Noise floor | High (verbose history) | Low (curated) |
| Storage growth | ~10x | baseline |
| Recall precision | Lower (noise dominates) | Higher |
| User trust | "Will it remember X?" yes | "Will it remember X?" maybe |
| Audit complexity | Hard (every turn) | Easy (snapshots are intentional) |

**Decision: keep explicit save as default. Ship opt-in auto-summarize mode in 0.8.0** (user decision 2026-05-16).

The new `PostToolUse` aggregator (3.4) gives us per-tool signal without bloating snapshots. For users who want claude-mem-style behavior, expose:

```yaml
# global or per-project config
memory:
  auto_summarize:
    enabled: false              # default: off, opt-in
    every_n_turns: 5            # roll-up every N assistant turns
    summarizer_capability: chat # capability route key (uses Azure or Ollama via existing factory)
    max_summary_chars: 500
    skip_when_idle: true        # don't fire if no Edit/Write/Bash since last summary
```

**Trigger model:** A new `Stop` hook handler counts assistant turns since the last `AutoSummary` snapshot for the session. When the counter reaches `every_n_turns` and `skip_when_idle` allows it, the handler:

1. Reads the last N turns from the transcript path passed in the hook envelope
2. Calls `config.create_llm_client(Capability::Chat)` to get the configured summarizer
3. Issues one chat completion with a deterministic prompt template (`"Summarize the following N-turn span into <= max_summary_chars chars, focusing on decisions made, files touched, and TODOs"`)
4. Writes the result as a snapshot with `trigger = AutoSummary` and `summarized_turn_range = (start, end)` metadata

**Cost controls:**

- Default `every_n_turns: 5` means at most 1 LLM call per 5 turns
- Falls back to deterministic template-based summarizer (no LLM) if `config.create_llm_client` errors (offline, no key, etc.)
- TUI badge distinguishes `AutoSummary` snapshots from `Explicit` ones; `clx prune --trigger auto_summary` lets users mass-delete

**New `SnapshotTrigger` variant:** add `AutoSummary` to `crates/clx-core/src/types.rs`. Migration `0009_auto_summary_trigger.sql` is a no-op (existing TEXT column accepts new value); type-system update in Rust is the only schema change.

**Insertion points (codebase map):**

- New file `crates/clx-hook/src/hooks/stop_auto_summary.rs` (~180 LoC)
- Wire in `crates/clx-hook/src/main.rs` router (in the refactored `handle_event`, §3.8)
- Storage helper `Storage::turns_since_last_auto_summary(session_id) -> Result<usize>`
- Summarizer module `crates/clx-core/src/summarize.rs` with `summarize_turns(turns, capability_route) -> Result<String>` (pure domain layer; calls infrastructure via existing `LlmBackend` trait)

This satisfies the user request for a claude-mem-style mode while keeping explicit save as the default. The opt-in nature is consistent with CLX's "low-noise by default" principle.

### 3.3 Pin Recent Sessions

**User ask:** "always inject the last N session summaries into UserPromptSubmit context, opt-in."

**Design:**

```yaml
auto_recall:
  pin_recent_sessions:
    enabled: false       # default off
    count: 3             # last N sessions
    max_chars_each: 300  # truncate per-summary
```

**Insertion point** (per codebase map):

- File: `crates/clx-hook/src/hooks/subagent.rs:135`
- After `debug!("Auto-recall: {} hits found", hits.len());`, before `format_recall_context(...)`, prepend pinned-session block:

```rust
let pinned = if config.auto_recall.pin_recent_sessions.enabled {
    storage
        .recent_session_summaries(config.auto_recall.pin_recent_sessions.count)
        .ok()
} else { None };

let combined = match pinned {
    Some(summaries) => format!("{}\n\n{}", format_pinned(&summaries), recall_body),
    None => recall_body,
};
```

- New storage method: `Storage::recent_session_summaries(n) -> Result<Vec<SessionSummary>>` (queries `sessions JOIN snapshots ON sessions.last_snapshot_id` for the most recent `n` rows where `summary IS NOT NULL`).

**Edge cases:**

- If fewer than N sessions exist: return what's available
- If a pinned session is the current session: skip it (no self-pinning)
- Output respects 500 ms budget by reading cached summaries only (no live LLM call)

### 3.4 PostToolUse Aggregator with `tool_events` Table

**Problem:** `events` table currently logs every tool call without dedup, leading to noise. The aggregator extracts signal: which mutating tools ran, against which files, with what outcome.

**Design:**

**Mutator-only filter** (whitelist):

```rust
const MUTATOR_TOOLS: &[&str] = &[
    "Edit", "Write", "NotebookEdit",
    "Bash",            // captured if command matches a mutator pattern (git commit, rm, cargo install)
    "MultiEdit",
];
```

Read-only tools (`Read`, `Grep`, `Glob`, `WebFetch`, MCP `clx_recall`, ...) are NOT aggregated.

**New table `tool_events`:**

```sql
CREATE TABLE tool_events (
  id INTEGER PRIMARY KEY,
  session_id TEXT NOT NULL,
  tool_name TEXT NOT NULL,
  target TEXT,                    -- file path or normalized command
  summary TEXT NOT NULL,          -- 1-line deterministic summary
  outcome TEXT NOT NULL,          -- 'success' | 'error'
  window_start_unix INTEGER NOT NULL,
  window_end_unix INTEGER NOT NULL,
  occurrence_count INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (session_id) REFERENCES sessions(id)
);

CREATE INDEX tool_events_session_idx ON tool_events (session_id, created_at DESC);
CREATE INDEX tool_events_target_idx ON tool_events (target);
```

**60s dedup window:**

When PostToolUse fires for a mutator:

1. Compute `target` (e.g., file path for Edit, normalized command for Bash)
2. Look up existing row where `tool_name = ?` AND `target = ?` AND `window_end_unix >= now - 60`
3. If found: increment `occurrence_count`, extend `window_end_unix = now`
4. If not: insert new row

**Deterministic summary template (no LLM):**

| Tool | Summary template |
|------|------------------|
| Edit | `edit <path> (+N -M lines)` |
| Write | `write <path> (N bytes)` |
| Bash | `bash <verb>: <truncated 80 char>` |
| NotebookEdit | `notebook <path> cell <idx>` |

**Recall weighting:**

When recall fetches context, `tool_events` rows contribute as auxiliary signal:

- `snapshot` rows: weight 0.7
- `tool_events` rows: weight 0.3 (only the most recent per target)

This means recent file mutations bubble up alongside snapshot summaries without dominating them.

**Retention** (user decision 2026-05-16: 30d default, configurable):

```yaml
retention:
  tool_events_days: 30      # default; set to 0 to disable trimming, or any positive int to override
  events_days: 7            # existing 'events' table (unchanged)
  snapshots_days: 0         # 0 = forever (unchanged)
```

A new `clx maintenance trim` command (also runnable as a background trim on `clx-hook` startup, gated by a 24h cooldown) runs `DELETE FROM tool_events WHERE created_at < datetime('now', ?)` with the configured window. Setting `tool_events_days: 0` skips the DELETE entirely (full history retained).

**Insertion point** (per codebase map):

- File: `crates/clx-hook/src/hooks/post_tool_use.rs:49`
- After existing `events` append, add `aggregate_tool_event(input, &storage)?` call
- New module: `crates/clx-core/src/storage/tool_events.rs` (~150 LoC)
- Migration: `migrations/0008_tool_events.sql`

### 3.5 Model-Discoverable Skills (6 skills, 2026 schema)

**Research finding:** Claude Code 2026 expects skills under `<plugin>/.claude-plugin/skills/<skill-name>/SKILL.md`. Current CLX layout has `plugin/skills/using-clx/SKILL.md` with one monolithic skill, which Claude Code 2026 will not discover.

**Anti-pattern to avoid:** "trigger bleed" (claude-mem issue #1651). Overly broad descriptions cause Claude to invoke skills when irrelevant. Each description must be:

- <= 1024 chars
- Starts with a specific user-facing trigger phrase
- Lists 2-3 concrete examples in the description

**Six skills to ship:**

| Skill name | Trigger surface | Body length target |
|------------|----------------|-------------------|
| `clx-recall` | "search past sessions", "what did I do last week" | ~80 LoC |
| `clx-remember` | "save this fact", "remember that" | ~60 LoC |
| `clx-checkpoint` | "create checkpoint before refactor" | ~50 LoC |
| `clx-rules` | "what rules apply", "refresh CLAUDE.md" | ~70 LoC |
| `clx-resume` | "resume earlier work on X" | ~90 LoC |
| `clx-doctor` | "why isn't recall working", "diagnose CLX" | ~120 LoC |

Each `SKILL.md` follows the same frontmatter contract:

```yaml
---
name: clx-recall
description: >
  Use when user references earlier work, past decisions, or asks
  "what did I do" about prior sessions. Invokes mcp__clx__clx_recall.
  Examples: "look up our discussion on Azure backend", "find checkpoints
  from yesterday", "what did we decide about retries".
---
```

**Migration plan:**

1. Move `plugin/plugin.json` -> `plugin/.claude-plugin/plugin.json`
2. Move `plugin/skills/using-clx/SKILL.md` -> archive (content split across 6 new skills)
3. Create 6 new skill directories under `plugin/.claude-plugin/skills/`
4. Update `plugin/scripts/validate.sh` to check 2026 schema (frontmatter, name length, description length)
5. Add `plugin/.claude-plugin/plugin.json` with updated `skills:` array referencing all 6

**Acceptance:** running Claude Code with the plugin installed must discover all 6 skills (verified by `claude /skills list`).

### 3.7 Dashboard Reducer Refactor

**Target:** `crates/clx/src/dashboard/event.rs` (373 LoC, 0% coverage).

**Current pattern (lines 12-37):** `run_event_loop` holds `&mut App`, mutates it inside the loop. Untestable without a full terminal.

**Elm Architecture refactor:**

```rust
pub enum DashboardEvent {
    Key(KeyEvent),
    RefreshTick,
    Resize(u16, u16),
    Quit,
}

pub fn update(state: AppState, event: DashboardEvent) -> (AppState, Vec<DashboardCmd>) {
    match event {
        DashboardEvent::Key(k) => handle_key(state, k),
        DashboardEvent::RefreshTick => (state.with_refresh_due(), vec![DashboardCmd::FetchData]),
        DashboardEvent::Resize(w, h) => (state.with_size(w, h), vec![]),
        DashboardEvent::Quit => (state.quit(), vec![]),
    }
}
```

`update` is pure: `(state, event) -> (state', cmds)`. The event loop becomes thin glue calling `update` and executing `cmds`.

**Test pattern:**

```rust
#[test]
fn quit_key_sets_should_quit() {
    let state = AppState::default();
    let (next, _) = update(state, DashboardEvent::Key(key('q')));
    assert!(next.should_quit);
}
```

This brings `event.rs` to high coverage without needing a TTY. The existing `App` struct is split into `AppState` (data, pure) and `AppRuntime` (terminal, mutex, etc.).

### 3.8 `clx-hook/main.rs` -> `handle_event` Refactor

**Target:** `crates/clx-hook/src/main.rs` (~190 LoC, 67% coverage).

**Extraction:**

```rust
pub struct HookDeps {
    pub config: clx_core::config::Config,
    pub storage: clx_core::storage::Storage,
}

pub async fn handle_event<R: Read, W: Write>(
    reader: R,
    writer: W,
    deps: HookDeps,
) -> ExitCode {
    let input = match read_and_parse(reader) {
        Ok(i) => i,
        Err(e) => return write_error(writer, e),
    };
    let result = route(input, deps).await;
    write_output(writer, result)
}

fn main() -> ExitCode {
    let deps = init_deps();        // tracing + config + storage
    let rt = tokio::runtime::Runtime::new().unwrap();
    rt.block_on(handle_event(io::stdin(), io::stdout(), deps))
}
```

`main()` shrinks to ~15 LoC of pure wiring (ignored by coverage); `handle_event` is fully testable with in-memory readers/writers.

**Bonus:** `assert_cmd` integration tests exercise the binary end-to-end with realistic JSON envelopes (3.10).

### 3.11 `clx trust` Command

**Problem:** Per-project `<repo>/.clx/config.yaml` is currently filtered to inert keys only. Power users want to opt-in to non-inert overrides (e.g., override `providers.azure.endpoint` for a project-specific Azure tenant).

**Design:**

```bash
clx trust add        # interactive: shows config hash, prompts confirmation, adds to trustlist
clx trust list       # shows trusted (hash, path, added_at)
clx trust remove <hash>
```

Trustlist stored at `~/.clx/trusted_configs.json`:

```json
[
  {
    "hash": "sha256:abc123...",
    "path": "/Users/x/repo/.clx/config.yaml",
    "added_at": "2026-05-15T10:00:00Z"
  }
]
```

**Behavior:**

- When loading `<repo>/.clx/config.yaml`:
  1. Compute SHA-256 of file contents
  2. If hash is in trustlist: apply full config (including non-inert keys)
  3. If not: apply only inert-key allowlist (current 0.7.x behavior)
- Any change to the file invalidates trust (hash changes); user must re-run `clx trust add`

**Insertion point** (per codebase map):

- File: `crates/clx-core/src/config/project.rs:86` (before `filter_value`)
- Branch: if trusted, return raw value; else continue with `filter_value`
- New module: `crates/clx-core/src/config/trust.rs` (~120 LoC)
- New CLI command: `crates/clx/src/commands/trust.rs`

**Security note:** Trust is per-machine, per-user, per-file-hash. It does not propagate via git. This is intentional: trusting a config is an explicit local action.
