### R1-F1 -- T9.2: silent allow of destructive commands (L0 off + L1 unreachable + `default_decision: allow`)

**Verdict:** **VULN-CONFIRMED**. Release-blocking. This is the F7-deferred
silent-allow class from v0.8.2; v0.9.0's new `layer0_enabled` toggle re-opens
the same bypass via a config-only path (no env var required).

**CVSS v4.0 vector (qualitative):** AV:L / AC:L / AT:N / PR:L / UI:N /
VC:H / VI:H / VA:H / SC:H / SI:H / SA:H. The attacker model is a hostile
*trusted* config (or any user who has been talked into setting these three
fields) plus a downed Ollama. Outcome: silent rubber-stamp of `rm -rf /`.

**SSVC:** Exploitation = PoC (immediate); Automatable = yes (config only);
Technical impact = total (validator bypass); Mission impact = irreversible
(arbitrary destructive command). Security-tool hard-override applies (any
validator bypass reachable without same-uid code-execution is
release-blocking).

**Counterexample (PoC):**
- `/Users/blackax/Projects/clx/crates/clx-hook/tests/v090_red_r1_poc.rs`
- Test: `t9_2_l0_off_l1_down_default_allow_destructive_command_is_silently_allowed`
- Config: `enabled=true, layer0_enabled=false, layer1_enabled=true,
  default_decision="allow", cache_enabled=false`, ollama host
  `http://127.0.0.1:1` (unreachable).
- Input envelope: `PreToolUse { tool=Bash, command="rm -rf /" }`.
- Output: `permissionDecision="allow"`.
- Audit DB: L0/`L0-DISABLED` row + L1/`allowed` fallback row. Both visible,
  but the **outcome was allow**.

**Run command:**
```

```

**Regression-pin (file:line, the surface a fix must touch):**
- `crates/clx-hook/src/hooks/pre_tool_use.rs:431-451` (LLM-client-error
  fallback emits `default_decision` without re-gating).
- `crates/clx-hook/src/hooks/pre_tool_use.rs:472-497` (LLM-unavailable
  fallback emits `default_decision` without re-gating).
- `crates/clx-hook/src/hooks/pre_tool_use.rs:518-544` (timeout fallback,
  same posture).
- `crates/clx-hook/src/hooks/pre_tool_use.rs:549-573` (LLM-generation-failure
  branch, same posture).
- `crates/clx-core/src/config/mod.rs:580-589` (config validation: refuse
  `default_decision=allow` when `layer0_enabled=false && layer1_enabled=true`?
  Or honour but warn-loud-and-block destructive patterns?).

**Recommended fix direction (RED does not implement):**
Apply the F7-deferred posture in v0.9.0 since the new toggle creates the
exact bypass class F7 was tracking. Three viable shapes (smallest correct
change wins):
1. **Refuse `default_decision=allow` when both layers are off OR when L0 is
   off and L1 is unreachable** -- fail-closed config validation at load.
2. **Force-ask on L1 unreachable when L0 was disabled** -- ignore
   `default_decision=allow` in this specific cross-product. The audit-row
   reason already records the combination so the override is forensic-grade.
3. **Loud-gate** -- if the operator opts into `default_decision=allow` with
   L0 off, require an explicit `i_understand_silent_allow: true` ack key.

**Residual-uncertainty:**
- The recon's "today's code: allow" claim is REPRODUCED above; no doubt.
- The fix shape is a posture call; (2) is smallest and most defensible.
- A user who genuinely opts into fail-open after reading the doc honestly
  may complain about (1). (2) is the recommendation.

### R1-F2 -- T1: SECURITY-CFG/SECURITY-ENV row tampering undetected by any in-tree validator

**Verdict:** **VULN-CONFIRMED**. Release-blocking only via doc-honesty
(T8); the underlying gap is "as designed" given the same-uid threat model,
but the **CHANGELOG / README claim "tamper-evident audit-chain
fingerprint" is misleading** in this context.

**CVSS v4.0:** AV:L / AC:L / PR:L / VC:N / VI:H / VA:N / SC:N / SI:H / SA:N.
Local same-uid attacker; integrity of the audit trail is compromised.
Confidentiality / availability unaffected (no exfiltration, no DoS).

**SSVC:** Exploitation = PoC; Automatable = yes (`sqlite3 INSERT`/`DELETE`);
Technical impact = partial (forensic trail); Mission impact = degraded.
Not release-blocking on its own under the same-uid threat model the rest
of the repo already accepts. **But the docs imply a stronger property**
than the code delivers; that is the release-blocking part (see T8).

**Counterexample (PoC):**

      a hand-crafted 64-hex `event_fingerprint=<64 'a's>`.
- (b) `Storage::get_audit_log_by_session(...)` reads it back as a genuine
      row -- no schema CHECK, no FK constraint on `layer`.
- (c) `sqlite3 <db> "DELETE FROM audit_log WHERE layer='SECURITY-CFG';"`
      succeeds (no triggers, no immutability, no append-only).
- Test: `t1_verify_fingerprint_sequence_is_dead_code_outside_tests` --

**Regression-pin:**
- `crates/clx-hook/src/audit_chain.rs:156-157` (`#[cfg_attr(not(test),
  allow(dead_code))]` -- the verifier exists but is never called in prod).
- `crates/clx-core/src/storage/migration.rs:188-200` (`audit_log` table

  user-supplied captured-warn anchor file. This wires
  `verify_fingerprint_sequence` to production.
- Hoist `event_fingerprint` into a first-class column (cheap migration v8)
  so a future operator can index and constrain it.

**Residual-uncertainty:**
- The "no production caller" check is grep-based and could be defeated by
  a future indirection (a re-export chain). The grep currently covers

### F1 -- redaction sinks: **VULN-REFUTED**

**Question:** when L0 is disabled and L1 fails, does the LLM-error path
(raw Azure body) still go through `redact_secrets` at every sink (azure.rs
error mapping + the 11 wrapped sinks)?

**Re-derived path:**
1. `pre_tool_use.rs:419-424` -- `Config::create_llm_client(Capability::Chat)`.
2. On client construction failure -> `pre_tool_use.rs:431-451`:
   `format!("Ollama client error: {e} -- default_decision: {}", ...)`. The
   `e` here is `clx_core::Error` Display, which wraps `LlmError::Connection(...)`.
3. `LlmError::Connection(String)` Display = `"connection failed: {0}"`
   (`crates/clx-core/src/llm/mod.rs:33-35`). The embedded String comes from:
   - `crates/clx-core/src/llm/azure.rs:285` -- `redact_secrets(&e.to_string())`
     **at construction**, BEFORE wrapping in `LlmError::Connection`.
4. The reasoning string is passed to `log_audit_entry`
   (`pre_tool_use.rs:444-447`), which at `crates/clx-hook/src/audit.rs:38`
   applies **`entry.reasoning = reasoning.map(redact_secrets)`** -- defense
   in depth at the sink.
5. Same defense applies on the LLM-unavailable, timeout, and
   generation-failure branches (`pre_tool_use.rs:472-497, 518-544, 549-573`).

**Counterexample (PoC):**
- Test: `f1_carryover_l0_off_l1_fail_redacts_provider_host_in_audit`.
- Config points ollama at synthetic `http://synthetic-tenant-xyz.openai.azure.com:1`
  (unreachable). Command `frobnicate --apply` (non-read-only). Asserts no
  L1 audit row contains the verbatim synthetic tenant hostname.

**Verdict:** v0.9.0 does NOT regress F1. Both the source-side redaction
(azure.rs:285) and the sink-side redaction (audit.rs:38) hold under the
new L0-disabled flow.

**Residual-uncertainty:** the synthetic tenant URL in the test is Azure-host
shaped (`*.openai.azure.com`) which the `redact_azure_hosts` boundary set
must catch. If a future provider uses a non-Azure hostname, the F1 fix
chain still relies on `redact_secrets` for the generic-secret patterns;
that is unchanged in v0.9.0.

## New findings beyond the recon hotlist

### R1-N1 -- `entry_fingerprint` is buried in free-form `reasoning TEXT`, not a column

**Verdict:** NEW-FINDING (cross-cutting, MEDIUM).

**Observation:** the v0.8.2 audit_chain reclassify and the v0.9.0
SECURITY-CFG extension both embed the SHA-256 hex fingerprint in the
`reasoning TEXT` column via `format!("...event_fingerprint={}", ...)`.
This is consistent with the recon's O1 / O2 observations but worth
calling out as a stand-alone finding because:
- Indexing on the fingerprint is impossible without parsing the string.
- A future `clx audit-verify` command must parse the reasoning text
  (brittle).
- A first-class column (`event_fingerprint TEXT, prev_fingerprint TEXT`)
  would enable proper verification queries and a schema CHECK constraint.

**Recommended fix direction:** v8 migration adds two columns;
`audit.rs::log_audit_entry` writes them; `verify_fingerprint_sequence`
reads them. Out of scope for v0.9.0 (would require a schema migration);
in-scope for v0.10.0.

### R1-N2 -- L0-DISABLED row writes the command verbatim before the read-only fast-lane (T10 confirmation + minor)

**Verdict:** NEW-FINDING (minor doc + intent). The recon flagged this as
T10. For a read-only command with L0 disabled, the audit DB receives
**two** rows: `L0/L0-DISABLED/{cmd}` AND `L0-READ/Allowed/{cmd}`. The
intent should be documented; today, the row asymmetry is observable but
not declared.

**Recommended fix direction:** add a one-line CHANGELOG note that the
L0-DISABLED row is per-command and may be paired with an L0-READ row;
no code change.

### R1-N3 -- `default_decision` parse via `#[derive(Default)]` is silent on intent

**Verdict:** NEW-FINDING (low). This is a v0.8.0 observation from
`specs/_prerelease/01-validation.md` that survives unchanged into v0.9.0.
The L0-toggle changes do NOT touch it, but the silent-allow class
(R1-F1) is sensitive to it: an operator who edits config without setting
`default_decision` explicitly gets whatever the enum's `Default` resolves
to. Best practice: pick `Ask` as the explicit `default = "ask"` serde
attribute and remove the implicit derive default. Out of scope for v0.9.0
unless GREEN ships R1-F1 fix shape (1) above.
