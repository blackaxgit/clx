## 1. Release-blocking class: silent-allow under L0-off + opt-in fail-open

The toggle introduces L0-disabled as a first-class supported configuration.
Combined with the pre-existing `default_decision=allow` opt-in, three
distinct failure paths produce a silent `allow` for the canonical
deterministic-blacklist command `rm -rf /`. The recon flagged this as
T9.2 (most-damaging missing test); R2 confirms via PoC plus extends it to
T9.3 (timeout) and T9.4 (generation-failed) â€” three independent fallback
branches with the same defect.

### 1.1 T9.2 (LLM unreachable)

**VERDICT:** VULN-CONFIRMED
**FILE:LINE:** `crates/clx-hook/src/hooks/pre_tool_use.rs:472-497`
**COUNTEREXAMPLE:**
```yaml
validator:
  enabled: true
  layer0_enabled: false
  layer1_enabled: true
  default_decision: "allow"
  cache_enabled: false
ollama:
  host: "http://127.0.0.1:1"   # dead loopback
  model: "x"
  max_retries: 0
```
Input: `{"tool_input":{"command":"rm -rf /"}}` -> output decision: `allow`.
Audit row L1/allowed with reasoning "Ollama unavailable â€” default_decision: allow".
**REGRESSION-PIN:** `pre_tool_use.rs:472-497` must NOT honour `allow`
when the command matches a hardcoded deny pattern; OR config-load time
must refuse `default_decision=allow` when `layer0_enabled=false`. The
F7-deferred posture from v0.8.2 should apply to v0.9.0 because this
toggle creates the exact bypass class F7 was tracking.
**RESIDUAL-UNCERTAINTY:** the user opted in via `default_decision=allow`;
RED claims the opt-in must be revoked when L0 is also off because the
combination is the literal worst case.
**PoC:** `red_t9_2_l0_off_l1_down_default_allow_silent_blacklist_allow`.

### 1.2 T9.3 (LLM timeout)

**VERDICT:** VULN-CONFIRMED
**FILE:LINE:** `pre_tool_use.rs:518-544`
**COUNTEREXAMPLE:** same config as 1.1 but `layer1_timeout_ms: 100` with
a wiremock that delays 5s. Decision: `allow`. Audit shows L1/allowed

**REGRESSION-PIN:** `pre_tool_use.rs:518-544`.
**PoC:** `red_t9_3_l0_off_l1_timeout_default_allow_silent_blacklist_allow`.

### 1.3 T9.4 (LLM generation failed)

**VERDICT:** VULN-CONFIRMED
**FILE:LINE:** `pre_tool_use.rs:549-573`
**COUNTEREXAMPLE:** same config as 1.1 but wiremock returns HTTP 500 on
`/api/generate`. `evaluate_with_llm` yields `Ask("LLM unavailable")`.

**REGRESSION-PIN:** `pre_tool_use.rs:549-573`.
**PoC:** `red_t9_4_l0_off_l1_gen_failed_default_allow_silent_blacklist_allow`.

### 1.4 T9.1 (cache HIT bypasses L0 deny)

**VERDICT:** VULN-CONFIRMED
**FILE:LINE:** `pre_tool_use.rs:368-396` (cache lookup AFTER L0-disabled
branch falls through).
**COUNTEREXAMPLE:** plant a poisoned cache row directly via Storage:
```rust
let cache_key = compute_cache_key("rm -rf /", "/tmp");
st.cache_decision(&cache_key, "allow", Some("poisoned"), Some(1), 3600);
```
Then invoke the hook with `layer0_enabled: false, layer1_enabled: true,
cache_enabled: true` and command `rm -rf /`. Decision: `allow`. The
L1-CACHE hit drives the output despite the L0 deny rule for the same
command being on the books.
**REGRESSION-PIN:** when L0 is disabled, the cache lookup at line 368
inherits the L0-deny-bypass property of the toggle. Either (a) cache
hits must not bypass a hardcoded deny-list even when L0 is off, or (b)
the L0-off toggle must invalidate all cache rows for L0-deny patterns
on entry (cache scrubbing), or (c) the cache layer must be disabled
implicitly when L0 is disabled.
**RESIDUAL-UNCERTAINTY:** requires a writeable Storage to plant the
poisoned cache. The PoC plants it via legitimate API; production
attackers can plant it by waiting for any LLM that previously emitted
`allow` for the command.
**PoC:** `red_t9_1_l0_off_cache_hit_for_blacklisted_cmd`.

## 3. MEDIUM findings

### 3.1 T2 â€” load_learned_rules runs unconditionally

**VERDICT:** VULN-CONFIRMED (behavioural; the disable promise leaks at
I/O level even though no security check is bypassed today)
**FILE:LINE:** `pre_tool_use.rs:265-270` (Storage::open_default +
load_learned_rules) BEFORE the L0 gate at `:275`.
**COUNTEREXAMPLE:** with `layer0_enabled: false`, the hook still opens
the SQLite DB and reads `learned_rules`. Observable side effects:
- `~/.clx/data/clx.db` exists and is populated after one invocation;
- migrations run on first call (cold-start cost);
- SQLite WAL lock contention with a concurrent `clx learn` writer
  is structurally possible (uncommon but defensible);
- if a future contributor inserts a "consult learned-rule whitelist
  before L1" hook between lines 365 and 399, the L0 disable is
  silently bypassed (maintenance hazard).

The PoC observes the DB exists + has audit rows after a single
L0-disabled invocation â€” proving the load runs and Storage was opened.

**REGRESSION-PIN:** `pre_tool_use.rs:265-270` should move inside the
`if config.validator.layer0_enabled { ... }` block at `:275`. OR the
property statement (the "L0 disabled = skip deterministic ruleset
entirely" comment at line 329-330) should be rewritten to say
"deterministic policy not consulted; learned-rule load still warms
the cache".
**RESIDUAL-UNCERTAINTY:** the disabled path performs additional disk
I/O (Storage::open_default + SQLite migration check + learned_rules
read) on every invocation; wall-clock impact is small (~ms) but
non-zero. The contention scenario requires a long-held WAL writer
lock from another CLX process.
**PoC:** `red_t2_l0_disabled_still_opens_storage`.

### 3.2 T3 â€” env + config double-audit on a single logical event

**VERDICT:** NEW-FINDING (the recon flagged the case; R2 confirms in
DB)
**FILE:LINE:** `pre_tool_use.rs:37-82` (SECURITY-ENV) AND `:93-128`
(SECURITY-CFG). Both fire when env and config both disable the same
layer.
**COUNTEREXAMPLE:** `CLX_VALIDATOR_LAYER0_ENABLED=false` env var AND
`validator.layer0_enabled: false` in config. One logical disable
event produces:
- one SECURITY-ENV row with reasoning containing
  `"CLX_VALIDATOR_LAYER0_ENABLED" + event_fingerprint=<hash_A>`,
- one SECURITY-CFG row with reasoning containing
  `"validator.layer0_enabled=false" + event_fingerprint=<hash_B>`,
- `hash_A != hash_B` (different trigger_keys -> different canonical
  serialization -> different SHA-256).

An aggregator deduping by fingerprint sees TWO events for ONE logical
disable. An aggregator deduping by `event_type` (always
`"validator_disabled"` in both) sees one. The CLX shipped behaviour
chose neither contract explicitly.

**REGRESSION-PIN:** `pre_tool_use.rs:93-128`. Either:
- (a) gate SECURITY-CFG with `&& !env_already_signalled_layer_disable(...)`
  so config-driven row is suppressed when env path fired (dedupe to one
  row, env wins), or
- (b) keep both and explicitly document the dual-signal contract in
  CHANGELOG + audit-chain docs (intentional defense-in-depth).

Per RED preference: (b) â€” the env vs config sources are different
threat models (CI/ops trust vs user config) and an operator may want
both signals. But the contract must be explicit and the fingerprints
must reflect a shared logical event id (e.g., bind both to a single
session-local epoch) so dedup is feasible.

**RESIDUAL-UNCERTAINTY:** downstream log-aggregator behaviour cannot be
enumerated here. The CLX docs ship no recommended aggregator config,
so any contract is implicit.
**PoC:** `red_t3_env_plus_config_double_audit`.
